BigQuery Fundamentals

1 What is BigQuery and how is it different from traditional RDBMS? ▼

BigQuery is Google Cloud's fully managed, serverless, petabyte-scale enterprise data warehouse designed for analytics (OLAP), not transactional workloads (OLTP).

AspectTraditional RDBMSBigQuery
ArchitectureCoupled compute + storage on a single server/clusterSeparated compute (Dremel) and storage (Colossus) — scales independently
Storage FormatRow-based (good for transactions)Columnar (Capacitor format — great for analytics)
ScalingVertical (bigger servers) or manual shardingAutomatic horizontal scaling — thousands of workers in seconds
InfrastructureYou manage servers, indexes, tuning, backupsZero infrastructure management — serverless
IndexingB-tree indexes required for fast readsNo indexes — uses columnar scans + partitioning + clustering
ConcurrencyRow-level locks, ACID transactionsSnapshot isolation, limited DML concurrency
PricingPer server/licensePer query (bytes scanned) or flat-rate slots
Best ForOLTP — frequent small reads/writesOLAP — large aggregations, analytics, reporting
💡 Key Takeaway: BigQuery is NOT a replacement for MySQL/PostgreSQL. It's purpose-built for scanning terabytes in seconds for analytics, not for single-row lookups or high-frequency transactional writes.
2 What is serverless in BigQuery? ▼

Serverless means you don't provision, configure, or manage any infrastructure. BigQuery handles everything behind the scenes:

  • No cluster management — no VMs to spin up, no Hadoop/Spark clusters to tune
  • Automatic scaling — BigQuery assigns compute resources (slots) dynamically based on query complexity
  • No capacity planning — storage grows automatically as you load data
  • No software patching — Google handles OS, security patches, engine upgrades
  • Pay-per-use — you pay for bytes scanned (on-demand) or reserved slots (capacity), not idle servers

What happens when you run a query:

  1. Your SQL goes to BigQuery's query engine (Dremel)
  2. Dremel creates an optimized execution plan and distributes it across thousands of workers
  3. Workers read data in parallel from Colossus (distributed storage)
  4. Results are aggregated and returned to you
  5. Workers are released — you never see or manage them
💡 Interview Angle: "Serverless doesn't mean no servers — it means the servers are Google's problem, not yours."

4 How does BigQuery store data internally? ▼

BigQuery uses a three-layer architecture for data storage:

  1. Colossus (Distributed File System) — Google's next-gen distributed file system (successor to GFS). All BigQuery data lives here as compressed, encrypted columnar files. Colossus handles replication, durability, and availability automatically.
  2. Capacitor (Columnar Format) — BigQuery's proprietary columnar storage format. Data is stored column-by-column rather than row-by-row. Each column is compressed independently using encoding schemes optimized for that column's data type (dictionary encoding, run-length encoding, etc.).
  3. Jupiter Network — Google's petabit-scale network fabric that connects compute (Dremel workers) to storage (Colossus) at extremely high throughput, enabling the separation of compute and storage without performance penalty.

How data is physically organized:

  • Tables are split into storage blocks (columnar chunks)
  • Each block stores data for a subset of rows, but in column-oriented layout
  • Metadata tracks min/max values per block for block pruning (skip irrelevant blocks)
  • All data is encrypted at rest (AES-256) and replicated across zones
💡 Key Point: Because compute (Dremel) and storage (Colossus) are separate, you can store petabytes cheaply and only pay for compute when you actually query.
5 What is columnar storage and why is it important? ▼

Columnar storage stores data column-by-column instead of row-by-row.

Row-based (Traditional RDBMS):

Row1: [id=1, name="Alice", age=30, city="NYC"]
Row2: [id=2, name="Bob",   age=25, city="LA"]
Row3: [id=3, name="Carol", age=35, city="NYC"]

Columnar (BigQuery):

id column:   [1, 2, 3]
name column: ["Alice", "Bob", "Carol"]
age column:  [30, 25, 35]
city column: ["NYC", "LA", "NYC"]

Why it matters for analytics:

  • Column pruning — SELECT name, age FROM t reads only 2 columns, skips the rest. In row-based storage, the entire row must be read.
  • Better compression — similar values in a column compress much better (e.g., "NYC" repeated → dictionary encoding). Typically 10x better compression than row storage.
  • Vectorized processing — CPU can process a batch of same-type values using SIMD instructions, much faster than processing mixed-type rows.
  • Cost savings — BigQuery charges per bytes scanned. Fewer columns read = less cost.
⚠️ This is why SELECT * is expensive in BigQuery — it reads ALL columns. Always select only the columns you need.

7 What is a dataset vs table vs view? ▼

BigQuery organizes data in a three-level hierarchy:

Project (GCP Project)
  └── Dataset (like a database/schema)
        ├── Table (actual data)
        ├── View (saved SQL query)
        ├── Materialized View (cached query result)
        └── Routine (UDF / stored procedure)
ConceptWhat It IsKey Details
DatasetA container for tables, views, routines • Equivalent to a schema in traditional RDBMS
• Has a geographic location (US, EU, etc.) — set at creation, can't change
• IAM permissions are set at dataset level
• Referenced as: project.dataset
TableStores actual data in columnar format • Types: Native (managed), External (GCS/Drive), Temporary
• Can be partitioned and clustered
• Supports schema evolution (add columns, relax REQUIRED → NULLABLE)
• Referenced as: project.dataset.table
ViewA saved SQL query — no data stored • Runs the underlying query every time it's accessed
• Can be authorized to share data without exposing source tables
• No extra storage cost, but full query cost each time
Materialized ViewA cached, pre-computed query result • BigQuery auto-refreshes when base table changes
• Zero-maintenance caching — optimizer can rewrite queries to use it
• Has storage cost (stores the result), but reduces query cost & latency
• Limited SQL support (no JOINs across tables in some cases)
💡 Best Practice: Use materialized views for expensive aggregations that are queried frequently (e.g., daily summary dashboards). Use regular views for access control (authorized views) and reusable query logic.
💡
Tip: BigQuery uses Dremel execution engine and Capacitor columnar format. It separates compute from storage, allowing massive parallelism without provisioning servers.

Partitioning & Clustering (VERY IMPORTANT)

1 What is table partitioning in BigQuery? ▼

Partitioning divides a large table into smaller, manageable segments (partitions) based on a column's value. When you query with a filter on the partition column, BigQuery only scans the relevant partitions — drastically reducing bytes processed and cost.

Three types of partitioning:

TypeHow It WorksUse Case
Ingestion-timeAutomatically partitions by _PARTITIONTIME or _PARTITIONDATE (when data was loaded)When you don't control the source schema
Time-unit columnPartition by a DATE, TIMESTAMP, or DATETIME column (HOUR, DAY, MONTH, YEAR granularity)Most common — event dates, transaction dates
Integer rangePartition by an integer column with start, end, and intervalCustomer IDs, zip codes, numeric keys
CREATE TABLE project.dataset.orders (
  order_id INT64,
  order_date DATE,
  amount NUMERIC
)
PARTITION BY order_date;  -- daily partitions by default
💡 Limit: Maximum 4,000 partitions per table. For daily partitions, that's ~11 years of data.
2 Difference between ingestion-time and column-based partitioning. ▼
AspectIngestion-Time PartitioningColumn-Based Partitioning
Partition KeyAuto-generated pseudo-column _PARTITIONTIMEA real column in your table (DATE/TIMESTAMP/INT)
ControlPartitioned by when data is loaded, not the data contentYou choose which column to partition on
Late-arriving dataGoes to the partition of load time, not event time → can cause incorrect partition placementGoes to the correct partition based on column value
GranularityDaily only (hourly since 2020)HOUR, DAY, MONTH, or YEAR
QueryingFilter with WHERE _PARTITIONDATE = '2026-01-01'Filter with WHERE order_date = '2026-01-01'
RecommendationLegacy — avoid for new tablesPreferred — more accurate and flexible
⚠️ Best Practice: Always prefer column-based partitioning. Ingestion-time partitioning was useful before column-based was available, but is now considered legacy.
3 What is clustering and how is it different from partitioning? ▼

Clustering sorts data within each partition (or the whole table) by up to 4 columns. BigQuery stores metadata about the min/max values in each data block, so queries filtering on clustered columns can skip irrelevant blocks.

FeaturePartitioningClustering
How it worksPhysically separates data into distinct partitionsSorts data within partitions/table by column values
Max columns1 column onlyUp to 4 columns
Column typesDATE, TIMESTAMP, DATETIME, INT64Any data type (STRING, INT, FLOAT, etc.)
Pruning typePartition pruning — entire partitions skippedBlock pruning — individual blocks within partition skipped
Cost estimateExact bytes shown before query runs (dry run)Estimate only — actual bytes may be less after block pruning
MaintenanceNone neededBigQuery auto-reclusters in background
CREATE TABLE project.dataset.orders (
  order_id INT64,
  order_date DATE,
  customer_id INT64,
  region STRING,
  amount NUMERIC
)
PARTITION BY order_date
CLUSTER BY region, customer_id;  -- sort within each partition
💡 Rule of Thumb: Partition for coarse-grained filtering (date ranges). Cluster for fine-grained filtering within partitions (specific IDs, categories, regions).
4 When should you use clustering instead of partitioning? ▼

Use clustering alone (without partitioning) when:

  • No suitable DATE/TIMESTAMP column — if your main filter is on STRING or high-cardinality INT columns
  • Too many partitions — partitioning on a column that would create >4,000 unique values
  • Partition sizes are too small — if each partition has <1 GB, partitioning overhead exceeds benefits. Clustering works better on smaller granularities
  • Multiple filter columns — you frequently filter on 2-4 columns together (e.g., region + product_type + status)

Use both together when:

  • You have a date column for coarse filtering AND other columns for fine filtering
  • Example: PARTITION BY event_date CLUSTER BY user_id, country
💡 Column Order Matters in Clustering: The first cluster column has the strongest sort. Place the most frequently filtered column first. CLUSTER BY region, customer_id is different from CLUSTER BY customer_id, region.
5 What happens if you don't filter on the partition column? ▼

Full table scan — BigQuery scans ALL partitions, which means:

  • You pay for scanning the entire table (all bytes across all partitions)
  • The query is slower because it processes far more data than needed
  • You get no benefit from partitioning at all

Common mistakes that cause full partition scans:

  • Using a function on the partition column: WHERE EXTRACT(YEAR FROM order_date) = 2026 — this prevents partition pruning!
  • Filtering on a different column than the partition key
  • Using subqueries/CTEs that obscure the partition filter
  • Dynamic filters like WHERE order_date = (SELECT MAX(date) FROM ...) — may not enable pruning
-- ❌ BAD: Function on partition column — scans ALL partitions
WHERE EXTRACT(YEAR FROM order_date) = 2026

-- ✅ GOOD: Direct filter — prunes to relevant partitions only
WHERE order_date BETWEEN '2026-01-01' AND '2026-12-31'
⚠️ Pro Tip: Enable require_partition_filter on critical tables to prevent accidental full scans: ALTER TABLE t SET OPTIONS (require_partition_filter = true)
6 Can a table be both partitioned and clustered? ▼

Yes! This is the recommended approach for most analytics tables. Partitioning provides coarse-grained pruning, and clustering provides fine-grained pruning within each partition.

CREATE TABLE project.dataset.events (
  event_id STRING,
  event_ts TIMESTAMP,
  user_id INT64,
  event_type STRING,
  country STRING
)
PARTITION BY DATE(event_ts)           -- coarse: daily partitions
CLUSTER BY country, event_type;       -- fine: sorted within each partition

Query example that benefits from both:

SELECT COUNT(*)
FROM project.dataset.events
WHERE DATE(event_ts) = '2026-04-22'   -- partition pruning → scan 1 partition
  AND country = 'US'                   -- block pruning → skip non-US blocks
  AND event_type = 'purchase';         -- further block pruning

8 How do you identify hot partitions? ▼

A hot partition is one that receives disproportionately more writes or is queried far more than others, causing performance bottlenecks.

How to detect:

-- Check partition sizes to find imbalanced partitions
SELECT
  table_name,
  partition_id,
  total_rows,
  total_logical_bytes / (1024*1024*1024) AS size_gb
FROM `project.dataset.INFORMATION_SCHEMA.PARTITIONS`
WHERE table_name = 'your_table'
ORDER BY total_logical_bytes DESC
LIMIT 20;

Signs of hot partitions:

  • One partition is 10x+ larger than others
  • Streaming inserts or DML always target the same partition (e.g., "today")
  • Queries on a specific partition are disproportionately slow

Mitigation strategies:

  • Switch from DAY to HOUR partitioning to distribute writes
  • Add clustering to improve read performance within hot partitions
  • Use MONTH or YEAR partitioning if daily creates too many small partitions
  • For streaming, use the BigQuery Storage Write API which handles hot partitions better than legacy streaming
ℹ️
Key Fact: BigQuery supports up to 4,000 partitions per table. Clustering supports up to 4 columns. Use partitioning for coarse filtering (e.g., date) and clustering for fine-grained filtering within partitions.

Query Performance & Cost Optimization

1How do you reduce query cost in BigQuery?▼

Top cost-reduction strategies (ranked by impact):

  1. Select only needed columns — avoid SELECT *. In columnar storage, each column is a separate scan cost.
  2. Use partition filters — always filter on the partition column to prune entire partitions.
  3. Use clustering — cluster on frequently filtered columns for block-level pruning.
  4. Use materialized views — pre-compute expensive aggregations; BigQuery auto-serves from cache.
  5. Avoid repeated CTEs — BigQuery executes CTEs every time they're referenced. Use temp tables for reuse.
  6. Use APPROX_* functions — APPROX_COUNT_DISTINCT is ~10x faster than COUNT(DISTINCT) for large cardinalities.
  7. Limit the result set — LIMIT doesn't reduce scan cost, but ordering large results is expensive.
  8. Use BI Engine — in-memory analysis acceleration for repeated dashboard queries.
  9. Set custom cost controls — maximum_bytes_billed on queries to prevent runaway costs.
-- Set a cost cap: query fails if it would scan more than 10 GB
SELECT col1, col2 FROM table
WHERE partition_date = '2026-04-22'
OPTIONS (maximum_bytes_billed = 10737418240);  -- 10 GB
2What does "bytes processed" mean?▼

Bytes processed is the total amount of data BigQuery reads from storage to execute your query. It is the primary billing metric for on-demand pricing ($6.25/TB).

What affects bytes processed:

  • Columns selected — each column scanned adds its compressed-on-disk size
  • Partitions scanned — without partition filter, all partitions are read
  • Table size — larger tables = more bytes

What does NOT count:

  • Cached results (0 bytes if query hits cache)
  • Dry run queries (no execution, just estimates)
  • Queries on INFORMATION_SCHEMA views
💡 Dry Run: Always use bq query --dry_run or the BigQuery UI's validator to check bytes before running expensive queries.
3Why is SELECT * bad in BigQuery?▼

Because BigQuery uses columnar storage, each column is stored and scanned independently. SELECT * forces BigQuery to read every column in the table.

Example on a 1 TB table with 50 columns:

  • SELECT * → scans all 50 columns → 1 TB scanned → $6.25
  • SELECT col1, col2 → scans 2 columns → ~40 GB scanned → $0.25

That's a 25x cost difference for the same number of rows.

⚠️ Exception: SELECT * EXCEPT(sensitive_col) is a BigQuery feature that selects all columns except the ones listed — useful but still scans most columns.
4How does predicate pushdown work?▼

Predicate pushdown is an optimization where filter conditions (WHERE clauses) are pushed as close to the storage layer as possible, so data is filtered before it reaches the compute layer.

How it works in BigQuery:

  1. Partition pruning — WHERE on partition column eliminates entire partitions at metadata level
  2. Block pruning (clustering) — min/max metadata on clustered column blocks allows skipping irrelevant blocks
  3. Column pruning — only requested columns are read from Capacitor files

What breaks predicate pushdown:

  • Wrapping partition column in a function: WHERE YEAR(date_col) = 2026
  • Using OR across different partitions with complex expressions
  • Filtering on computed/derived columns in CTEs
5What is slot contention?▼

Slot contention occurs when multiple queries compete for the same pool of compute slots, causing queries to queue or run slower.

Symptoms:

  • Queries that normally take 5 seconds now take 30+ seconds
  • Query execution plan shows stages waiting for slot allocation
  • INFORMATION_SCHEMA.JOBS shows high total_slot_ms relative to wall time

Solutions:

  • Reservations — purchase dedicated slots via BigQuery Editions
  • Workload management — assign different projects to different reservation pools
  • Optimize queries — reduce slot consumption by limiting JOINs, using APPROX functions
  • Schedule jobs — stagger ETL pipelines to avoid peak contention windows
6How does BigQuery optimize joins?▼

BigQuery's optimizer automatically chooses the best join strategy:

StrategyHow It WorksWhen Used
Broadcast JoinSmaller table is copied to every worker processing the larger tableOne table is small enough to fit in memory (~10 MB default threshold)
Hash Join (Shuffle)Both tables are shuffled/redistributed by join key across workersBoth tables are large — data must be co-located by key
Sort-Merge JoinBoth tables are sorted by join key, then mergedLarge-to-large joins on sorted (clustered) data

Optimization tips:

  • Always put the larger table on the left side of the JOIN
  • Filter both tables before the JOIN to reduce shuffle data
  • If one table is small, use SELECT ... FROM big_table JOIN small_table — BigQuery auto-broadcasts
  • Avoid CROSS JOIN unless absolutely necessary

8What is the use of APPROX_* functions?▼

BigQuery provides approximate aggregation functions that trade a small accuracy margin (typically <1%) for massive performance gains:

FunctionReplacesUse Case
APPROX_COUNT_DISTINCT(col)COUNT(DISTINCT col)Cardinality estimation (unique users, sessions)
APPROX_QUANTILES(col, n)Exact percentile calculationMedian, P95, P99 computations
APPROX_TOP_COUNT(col, n)GROUP BY + ORDER BY + LIMITTop N most frequent values
APPROX_TOP_SUM(col, weight, n)Weighted top NTop N by revenue, clicks

Performance: APPROX_COUNT_DISTINCT uses HyperLogLog++ algorithm — runs up to 10x faster and uses far less memory than exact COUNT(DISTINCT) on high-cardinality columns (millions of unique values).

💡 When to use: Dashboards and analytics where ~99% accuracy is acceptable. When NOT to use: Financial reporting, billing, or any use case requiring exact counts.
9How do you analyze query execution details?▼

Three main tools:

  1. Query Execution Plan (Explanation) — visible in BigQuery UI after running a query. Shows stages, slots used, shuffle bytes, read/write ratios.
  2. INFORMATION_SCHEMA.JOBS — query job metadata programmatically:
SELECT
  job_id,
  total_bytes_processed / (1024*1024*1024) AS gb_scanned,
  total_slot_ms / 1000 AS slot_seconds,
  TIMESTAMP_DIFF(end_time, start_time, SECOND) AS wall_seconds,
  cache_hit
FROM `region-us`.INFORMATION_SCHEMA.JOBS
WHERE creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
ORDER BY total_bytes_processed DESC
LIMIT 20;
  1. Execution plan key metrics to watch:
  • Slot time vs wall time ratio — high ratio = good parallelism
  • Shuffle bytes — high shuffle = large JOINs or GROUP BYs redistributing data
  • Spill to disk — indicates insufficient memory during aggregation
  • Input/output row ratio — stages reading many rows but outputting few = filter is working well
⚠️
Cost Trap: SELECT * scans all columns. In columnar storage, only selecting needed columns can reduce cost by 90%+. Always use column pruning and partition filters.

2How do you UNNEST arrays?▼

UNNEST() converts an ARRAY into rows — essentially a cross join that expands each array element into its own row.

-- Simple UNNEST
SELECT val
FROM UNNEST([10, 20, 30]) AS val;
-- Result: 3 rows → 10, 20, 30

-- UNNEST with parent table (correlated)
SELECT
  o.order_id,
  item.product,
  item.qty
FROM `orders` o,
UNNEST(o.items) AS item;    -- items is ARRAY<STRUCT>

-- LEFT JOIN UNNEST (keep rows even if array is empty)
SELECT o.order_id, item.product
FROM `orders` o
LEFT JOIN UNNEST(o.items) AS item;
💡 Key Difference: FROM table, UNNEST(col) is an implicit CROSS JOIN — rows with empty arrays are dropped. Use LEFT JOIN UNNEST to preserve them.
3Difference between SAFE_CAST and CAST.▼
CASTSAFE_CAST
On failureQuery fails with errorReturns NULL
Use whenData is clean and should always cast successfullyData may have bad values (dirty columns, mixed types)
-- CAST fails on bad data
SELECT CAST("abc" AS INT64);  -- ERROR!

4What is ANY_VALUE() and when should you use it?▼

ANY_VALUE(col) returns an arbitrary value from the group. It tells BigQuery: "I know all values in this group are the same, just pick one."

When to use:

  • You're grouping by a key, and a non-aggregated column has the same value for all rows in each group
  • Without ANY_VALUE, BigQuery forces you to either add it to GROUP BY or use an aggregate like MAX
-- Problem: user_name isn't in GROUP BY
SELECT user_id, user_name, COUNT(*) AS orders   -- ERROR!
FROM orders GROUP BY user_id;

-- Fix 1: ANY_VALUE (cleanest when name is same per user)
SELECT user_id, ANY_VALUE(user_name) AS user_name, COUNT(*) AS orders
FROM orders GROUP BY user_id;

-- Fix 2: Add to GROUP BY (works but can affect performance)
SELECT user_id, user_name, COUNT(*) AS orders
FROM orders GROUP BY user_id, user_name;
💡 Performance: ANY_VALUE avoids adding columns to GROUP BY, which can reduce shuffle and improve performance on large datasets.
5What is SELECT * EXCEPT?▼

A BigQuery-specific syntax that selects all columns except the ones you list. Useful when a table has many columns and you want to exclude a few.

-- Select everything except sensitive columns
SELECT * EXCEPT(ssn, password_hash)
FROM users;

-- Combined with REPLACE to transform a column
SELECT * REPLACE(UPPER(name) AS name)
FROM users;

-- Both together
SELECT * EXCEPT(temp_col) REPLACE(ROUND(amount, 2) AS amount)
FROM transactions;
⚠️ Note: SELECT * EXCEPT still scans all remaining columns. It doesn't help with cost as much as explicitly listing only needed columns.
6How does QUALIFY work?▼

QUALIFY filters the result of window functions — just like HAVING filters aggregations. It eliminates the need for a subquery.

-- WITHOUT QUALIFY (subquery needed)
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) AS rn
  FROM events
) WHERE rn = 1;

-- WITH QUALIFY (clean, no subquery)
SELECT *
FROM events
QUALIFY ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) = 1;

SQL execution order with QUALIFY:

FROM → WHERE → GROUP BY → HAVING → WINDOW → QUALIFY → ORDER BY → LIMIT
💡 Common Pattern: Deduplication — QUALIFY ROW_NUMBER() OVER (PARTITION BY key ORDER BY ts DESC) = 1 keeps the latest record per key.

  • ROW_NUMBER() — always unique, arbitrary tiebreak. Best for deduplication.
  • RANK() — same value = same rank, then skips. "1st, 2nd, 2nd, 4th".
  • DENSE_RANK() — same value = same rank, no skip. "1st, 2nd, 2nd, 3rd".
💡 Interview tip: Always specify what you need: Use ROW_NUMBER() for dedup, RANK() for competition ranking, DENSE_RANK() for top-N with ties.
8How do you handle NULLs in BigQuery?▼

BigQuery follows the SQL standard for NULLs. Key behaviors:

  • NULL = NULL → NULL (not TRUE!) — use IS NULL / IS NOT NULL
  • NULL in aggregation — most aggregate functions ignore NULLs (SUM, AVG, COUNT(col))
  • COUNT(*) counts all rows; COUNT(col) skips NULLs

Useful NULL-handling functions:

IFNULL(col, default)       -- returns default if col is NULL
COALESCE(a, b, c)          -- returns first non-NULL value
NULLIF(a, b)               -- returns NULL if a = b, else a
IF(condition, val, NULL)   -- conditional NULL
SAFE_DIVIDE(a, b)          -- returns NULL instead of division-by-zero error
💡 NULL in JOINs: NULL values never match in JOIN conditions. If you need to join on nullable columns, use IFNULL(col, sentinel) or COALESCE.
💡
QUALIFY filters window function results without a subquery: SELECT * FROM t QUALIFY ROW_NUMBER() OVER (PARTITION BY id ORDER BY ts DESC) = 1

BigQuery Architecture & Internals

1How does BigQuery handle scalability?▼

BigQuery achieves massive scalability through separation of compute and storage and multi-tenant architecture:

  • Storage (Colossus) — scales independently to petabytes. You pay $0.02/GB/month for active storage, $0.01/GB for long-term (>90 days untouched).
  • Compute (Dremel) — dynamically allocates thousands of workers per query. A single query can use 2,000+ slots in parallel.
  • Network (Jupiter) — 1 Pb/s bisection bandwidth between compute and storage, so separation doesn't create a bottleneck.
  • Multi-tenancy — on-demand users share a massive global slot pool. Google manages utilization across millions of users.
  • Automatic sharding — data is automatically split into optimal-sized blocks across Colossus. No manual sharding needed.
💡 Key Insight: BigQuery can scale from scanning 1 MB to 1 PB without any configuration change. You don't resize clusters — you just run the query.
2What is Dremel?▼

Dremel is Google's internal query execution engine — the core technology behind BigQuery.

Key characteristics:

  • Tree architecture — queries are distributed through a serving tree: root server → intermediate servers → leaf servers (workers)
  • Columnar execution — operates directly on compressed columnar data without decompressing entire rows
  • In-situ querying — reads data where it lives (Colossus/GCS) without requiring data to be imported first
  • Multi-level execution — each tree level aggregates partial results, reducing data movement
            Root Server
           /    |    \
      Mixer   Mixer   Mixer      ← intermediate aggregation
      / \      / \      / \
    Leaf Leaf Leaf Leaf Leaf Leaf ← read data from Colossus
💡 History: Dremel was published in a 2010 Google research paper. It inspired Apache Drill and other distributed SQL engines.
3What is Capacitor?▼

Capacitor is BigQuery's proprietary columnar storage format — think of it as Google's equivalent to Apache Parquet or ORC, but optimized for BigQuery's architecture.

Key features:

  • Column-oriented — each column stored separately for efficient scanning
  • Encoding per column — dictionary encoding, run-length encoding, delta encoding chosen automatically per column's data characteristics
  • Nested data support — natively handles STRUCT and ARRAY types using definition/repetition levels (from Dremel paper)
  • Block metadata — stores min/max values, null counts per block for partition/block pruning
  • Encryption — all data encrypted at rest with AES-256 by default
💡 You never interact with Capacitor directly. It's an internal format. Data is automatically converted to Capacitor on ingestion (from CSV, JSON, Avro, Parquet, etc.).
4How does BigQuery manage fault tolerance?▼

BigQuery provides fault tolerance at multiple layers:

  • Storage (Colossus) — data is automatically replicated across multiple zones and data centers. No single point of failure.
  • Compute (Dremel) — if a worker fails mid-query, the scheduler reassigns that worker's task to another worker. The query continues without the user noticing.
  • Checkpointing — long-running queries checkpoint intermediate results, so a failure doesn't restart from scratch.
  • Shuffle persistence — shuffle data (intermediate results between stages) is written to a persistent shuffle service, surviving individual worker failures.
  • Regional redundancy — multi-region datasets are replicated across geographic regions for disaster recovery.
💡 SLA: BigQuery offers 99.99% uptime SLA for multi-region deployments and 99.9% for single-region.
5How does BigQuery handle schema changes?▼

BigQuery supports limited schema evolution without downtime:

6What is time travel in BigQuery?▼

Time travel lets you query a table's data at any point within the last 7 days (configurable from 2 to 7 days), even if data was modified or deleted.

-- Query table as it was 1 hour ago
SELECT * FROM `project.dataset.table`
FOR SYSTEM_TIME AS OF TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 HOUR);

-- Query table at a specific timestamp
SELECT * FROM `project.dataset.table`
FOR SYSTEM_TIME AS OF '2026-04-21 10:00:00 UTC';

-- Restore accidentally deleted data
INSERT INTO `project.dataset.table`
SELECT * FROM `project.dataset.table`
FOR SYSTEM_TIME AS OF TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 MINUTE);

Use cases: undo accidental DELETEs/UPDATEs, audit data changes, debug pipeline issues, restore tables.

⚠️ Important: Time travel has a storage cost — BigQuery keeps changed/deleted data for the time travel window. Configure with: ALTER TABLE t SET OPTIONS (max_staleness = INTERVAL 3 DAY)
7What are table snapshots?▼

A table snapshot is a read-only, point-in-time copy of a table. It captures the table's state at a specific moment.

-- Create snapshot of current state
CREATE SNAPSHOT TABLE `project.dataset.orders_snapshot`
CLONE `project.dataset.orders`;

-- Create snapshot at a specific time (within time travel window)
CREATE SNAPSHOT TABLE `project.dataset.orders_snap_april20`
CLONE `project.dataset.orders`
FOR SYSTEM_TIME AS OF '2026-04-20 00:00:00 UTC';

Properties:

  • Read-only — cannot be modified
  • Minimal storage cost — only stores metadata + deltas from the base table
  • Can be used to restore tables: CREATE OR REPLACE TABLE orders CLONE orders_snapshot
  • Retained beyond the 7-day time travel window

BigQuery Data Modeling

1When should you use denormalization in BigQuery?▼

Almost always. Unlike traditional RDBMS where normalization reduces redundancy, BigQuery favors denormalized (flat/wide) tables because:

  • JOINs are expensive — they require shuffling large datasets across workers
  • Columnar storage handles wide tables efficiently — unused columns aren't scanned
  • Storage is cheap ($0.02/GB/month) — redundancy costs less than repeated JOIN compute
  • Nested/repeated fields (STRUCT + ARRAY) let you denormalize without full flattening

When to keep normalized:

  • Dimension tables that change frequently (avoid updating across many denormalized rows)
  • Very small lookup tables (broadcast joins are cheap)
-- Instead of: orders JOIN customers JOIN products (3-way join)
-- Denormalize into a single wide table:
CREATE TABLE analytics.order_facts AS
SELECT
  o.order_id, o.order_date, o.amount,
  c.customer_name, c.country,        -- denormalized from customers
  p.product_name, p.category          -- denormalized from products
FROM orders o
JOIN customers c USING(customer_id)
JOIN products p USING(product_id);
2How do you model event-based data?▼

Event data (clickstreams, transactions, logs) is naturally append-only. Recommended approach:

  • Partition by event date — PARTITION BY DATE(event_timestamp)
  • Cluster by high-cardinality filters — CLUSTER BY user_id, event_type
  • Nest event properties — use STRUCT for event metadata, ARRAY<STRUCT> for sub-events
  • Use streaming inserts or Storage Write API for real-time ingestion
  • Set table expiration for short-lived staging events
CREATE TABLE events (
  event_id STRING,
  event_ts TIMESTAMP,
  user_id INT64,
  event_type STRING,
  properties STRUCT<
    page STRING,
    referrer STRING,
    duration_ms INT64
  >,
  tags ARRAY<STRING>
)
PARTITION BY DATE(event_ts)
CLUSTER BY user_id, event_type;
3How do you handle slowly changing dimensions?▼

SCD (Slowly Changing Dimensions) in BigQuery — common strategies:

TypeStrategyBigQuery Implementation
SCD1 (Overwrite)Always keep latest valueMERGE ... WHEN MATCHED THEN UPDATE
SCD2 (History)Keep full history with effective datesAdd effective_from, effective_to, is_current columns. Use MERGE to close old record and insert new one.
SCD3 (Previous value)Store current + previous valueAdd prev_value column. UPDATE current, shift old to prev.
SnapshotFull snapshot per dayPartition by snapshot_date. Simplest but storage-heavy.
-- SCD2 with MERGE
MERGE dim_customer T
USING staging S ON T.customer_id = S.customer_id AND T.is_current = TRUE
-- Close existing record
WHEN MATCHED AND T.address != S.address THEN
  UPDATE SET T.effective_to = CURRENT_DATE(), T.is_current = FALSE
-- Insert new current record (done separately after closing)
WHEN NOT MATCHED THEN
  INSERT (customer_id, address, effective_from, effective_to, is_current)
  VALUES (S.customer_id, S.address, CURRENT_DATE(), DATE '9999-12-31', TRUE);
💡 BigQuery preference: Daily snapshots (partition by date) are simplest and work well since storage is cheap. SCD2 is better when storage efficiency matters.
4How do you store nested JSON?▼

Two approaches:

  • Native nested types (STRUCT + ARRAY) — preferred. Schema is enforced, individual fields are scannable, and columnar compression works per field.
  • JSON STRING column — store raw JSON as string. Flexible but requires JSON_EXTRACT_SCALAR() at query time, no column pruning, poor performance.
-- Approach 1: STRUCT + ARRAY (recommended)
CREATE TABLE orders (
  order_id STRING,
  customer STRUCT<name STRING, email STRING>,
  items ARRAY<STRUCT<product STRING, qty INT64, price NUMERIC>>
);

-- Query nested fields directly
SELECT customer.name, item.product, item.price
FROM orders, UNNEST(items) AS item;

-- Approach 2: JSON string (flexible but slow)
CREATE TABLE raw_events (
  event_id STRING,
  payload STRING  -- raw JSON
);

SELECT JSON_EXTRACT_SCALAR(payload, '$.user.name') AS user_name
FROM raw_events;

Since 2022, BigQuery also supports the native JSON data type — a middle ground with semi-structured data support and dot-notation access.

6What are the limitations of materialized views?▼

Current limitations:

  • Single base table — can reference only one table (multi-table support is limited/preview)
  • Limited SQL — no UNION, no non-deterministic functions (CURRENT_TIMESTAMP()), no UDFs
  • Aggregation only — must include at least one aggregate function (COUNT, SUM, AVG, etc.)
  • No DML — cannot INSERT/UPDATE/DELETE directly on a materialized view
  • Max 20 per base table
  • Refresh lag — not guaranteed to be up-to-the-second fresh (typically minutes)
  • Storage cost — you pay for storing the materialized result
  • Streaming buffer — data in the streaming buffer may not be reflected immediately
⚠️ Workaround: If you need a multi-table materialized view, create a scheduled query (via BigQuery Scheduled Queries) that writes results to a table on a defined schedule.
💡

2What is the streaming buffer?▼

When you use streaming inserts, data first lands in a streaming buffer — a temporary, high-speed storage area within BigQuery.

Characteristics:

  • Data is immediately queryable after streaming
  • Buffer retains data for approximately 90 minutes before flushing to permanent columnar storage
  • Data in the buffer is not subject to DML operations (UPDATE/DELETE)
  • Shows as "Streaming buffer" in table details in the BigQuery UI
  • The buffer is NOT backed by Capacitor format — it's in an optimized write format
  • Table COPY and EXPORT operations may miss data in the buffer

Check streaming buffer status:

-- Check if table has data in streaming buffer
SELECT * FROM `project.dataset.INFORMATION_SCHEMA.TABLE_STORAGE`
WHERE table_name = 'my_table';
3Can you query data in the streaming buffer?▼

Yes — data is queryable within seconds of streaming. However, there are caveats:

  • SELECT queries — work normally, buffer data is included in results
  • DML (UPDATE/DELETE) — cannot modify data still in the buffer. Wait ~90 min or until it flushes.
  • Time travel — buffer data is NOT included in FOR SYSTEM_TIME AS OF queries
  • Table wildcard queries — table_* queries include buffer data
  • COPY/EXPORT — may miss buffer data. Wait for buffer to flush before exporting.
💡 With Storage Write API's committed mode, data is immediately available for all operations including DML — no buffer delay.
4How do you handle duplicate records in streaming data?▼

Strategies from simplest to most robust:

  1. insertId (legacy streaming) — pass a unique insertId per row. BigQuery does best-effort deduplication within the buffer (not guaranteed). Deprecated in favor of Storage Write API.
  2. Storage Write API — committed mode — provides exactly-once semantics with stream offsets. Recommended approach.
  3. Dedup at query time — use QUALIFY ROW_NUMBER() OVER (PARTITION BY id ORDER BY ts DESC) = 1 to pick one record per key.
  4. MERGE periodically — run a scheduled MERGE to deduplicate from a staging table to the final table.
  5. Dedup view — create a view with the QUALIFY dedup logic. Consumers always get clean data.
-- Dedup view pattern
CREATE VIEW clean_events AS
SELECT * FROM raw_events
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY event_id ORDER BY ingestion_ts DESC
) = 1;

Security & Governance

1What is row-level security?▼

Row-level security (RLS) restricts which rows a user can see based on their identity. Implemented using Row Access Policies.

-- Create a policy: users can only see rows where region matches their group
CREATE ROW ACCESS POLICY region_filter
ON `project.dataset.sales`
GRANT TO ('group:us-analysts@company.com')
FILTER USING (region = 'US');

-- Another policy for EU team
CREATE ROW ACCESS POLICY eu_filter
ON `project.dataset.sales`
GRANT TO ('group:eu-analysts@company.com')
FILTER USING (region = 'EU');

-- Admins see everything
CREATE ROW ACCESS POLICY admin_access
ON `project.dataset.sales`
GRANT TO ('group:data-admins@company.com')
FILTER USING (TRUE);

Key details:

  • Multiple policies per table — they combine with OR logic (user sees rows matching ANY of their policies)
  • If a user has NO matching policy, they see zero rows (deny by default)
  • Transparent to queries — users write normal SQL; filters are applied automatically
  • Works with all query types including JOINs, aggregations, exports
2What is column-level security?▼

Column-level security restricts access to specific columns using policy tags from Data Catalog.

How it works:

  1. Create a taxonomy in Data Catalog (e.g., "Data Classification")
  2. Create policy tags within the taxonomy (e.g., "PII", "Confidential", "Public")
  3. Assign policy tags to table columns (e.g., ssn → PII, salary → Confidential)
  4. Grant Fine-Grained Reader role to users who should see protected columns

Behavior:

  • Users without the Fine-Grained Reader role get an Access Denied error if they query a protected column
  • SELECT * also fails — user must explicitly exclude protected columns with SELECT * EXCEPT(ssn)
  • Works with authorized views — the view can access protected columns on behalf of the user
💡 Data masking (preview): Instead of full deny, you can configure data masking rules — users see masked values (e.g., ***-**-1234) instead of being blocked entirely.
3How do authorized views work?▼

An authorized view can query tables that the view's users don't have direct access to. The view acts as a security layer.

How to set up:

  1. Create a view in dataset B that queries a table in dataset A
  2. In dataset A, add the view as an authorized view (dataset settings → Authorized Views)
  3. Grant users access to the view (dataset B), but NOT to dataset A's tables
  4. Users can query the view → it accesses dataset A's data on their behalf

Use cases:

  • Share aggregated data without exposing raw tables
  • Filter rows/columns based on business logic in the view
  • Cross-dataset data sharing within an organization
💡 Authorized views bypass RLS? No — row access policies still apply. But the view can be used with column-level security to expose only permitted columns.

Scenario-Based Questions (Interview Favorites)

1A query suddenly becomes expensive — how do you debug?▼

Step-by-step debugging approach:

  1. Check bytes scanned — compare with historical runs in INFORMATION_SCHEMA.JOBS. Sudden increase = something bypassed pruning.
  2. Check execution plan — look for full partition scans, high shuffle bytes, or spill-to-disk stages.
  3. Verify partition filters — is the WHERE clause filtering on the partition column? Did a code change remove it?
  4. Check for schema changes — new columns added? SELECT * now scans more data.
  5. Check data volume — did the source table grow significantly? New data loaded?
  6. Check clustering — if data wasn't reclustered after major loads, block pruning degrades.
  7. Check for cross-join or Cartesian products — accidental cross joins explode row counts.
  8. Check slot utilization — high total_slot_ms relative to wall time suggests contention.
-- Compare today's query cost vs 7 days ago
SELECT
  DATE(creation_time) AS day,
  SUM(total_bytes_processed) / POW(1024,4) AS tb_scanned,
  AVG(total_slot_ms) / 1000 AS avg_slot_sec
FROM `region-us`.INFORMATION_SCHEMA.JOBS
WHERE query LIKE '%my_table%'
  AND creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
GROUP BY 1 ORDER BY 1;
2A partitioned table is scanning all partitions — why?▼

Common causes:

  1. No partition filter — query doesn't have a WHERE clause on the partition column
  2. Function on partition column — WHERE EXTRACT(YEAR FROM dt) = 2026 breaks pruning
  3. Subquery-based filter — WHERE dt = (SELECT MAX(dt) FROM ...) may not be optimized
  4. CTE obscures filter — partition filter applied after CTE materialization
  5. OR conditions — WHERE dt = '2026-01-01' OR user_id = 123 — the OR forces full scan
  6. Wrong column — filtering on a non-partition column that looks similar
  7. Dynamic table reference — using scripting/variables to build the filter

Prevention:

-- Force partition filter requirement
ALTER TABLE `project.dataset.my_table`
SET OPTIONS (require_partition_filter = true);
-- Now any query without a partition filter will FAIL
3How would you design a cost-efficient ingestion pipeline?▼

Design principles:

  • Use batch loads when possible — free! Schedule with Cloud Composer/Airflow
  • Land data in GCS first → load from GCS into BigQuery (decouple ingestion from processing)
  • Use Parquet/Avro format — compressed, columnar, faster than CSV/JSON
  • Partition by date on load — use --time_partitioning_field in bq load
  • Use Storage Write API for real-time needs — exactly-once, lower cost than legacy streaming
  • Separate raw/staging/curated layers — raw tables for ingestion, curated tables for analysis
Architecture:
Source → Cloud Pub/Sub → Dataflow → GCS (Parquet) → BQ Load Job → Raw Table
                                                                        ↓
                                              Scheduled Query → Curated Table (partitioned + clustered)
                                                                        ↓
                                                            Materialized View → Dashboard
4You need to join a 10 TB table with a 1 GB table — how?▼

BigQuery handles this automatically — the optimizer will use a broadcast join, sending the 1 GB table to all workers processing the 10 TB table.

Best practices for this scenario:

  • Put the large table on the LEFT — FROM big_table JOIN small_table
  • Filter before joining — reduce the 10 TB table with partition/where filters first
  • Verify in execution plan — check that the plan shows BROADCAST not HASH
  • Consider denormalization — if this join runs daily, pre-join the small table into the large one
-- Optimal: filter large table first, then join
SELECT o.*, c.customer_name
FROM (
  SELECT * FROM big_orders
  WHERE order_date BETWEEN '2026-04-01' AND '2026-04-22'  -- prune partitions
) o
JOIN small_customers c USING(customer_id);  -- auto broadcast
5How do you reprocess last 7 days of data efficiently?▼

Use partition-based overwrite to reprocess only the affected date range:

-- Step 1: Delete affected partitions
DELETE FROM `project.dataset.curated_table`
WHERE event_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY);

-- Step 2: Re-insert from raw data
INSERT INTO `project.dataset.curated_table`
SELECT
  event_date,
  -- transformation logic here
FROM `project.dataset.raw_table`
WHERE event_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY);

-- Alternative: MERGE for upsert semantics
MERGE curated T
USING (SELECT ... FROM raw WHERE event_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY)) S
ON T.id = S.id AND T.event_date = S.event_date
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT ...;
💡 Key: Always filter on the partition column in both DELETE and INSERT so only 7 partitions are touched, not the entire table.
6How do you archive old data cheaply?▼

Options ranked by cost-effectiveness:

  1. Do nothing — BigQuery auto-moves untouched data to long-term storage after 90 days ($0.01/GB/month vs $0.02/GB for active). The cheapest option if you still need queryability.
  2. Export to GCS + delete from BigQuery — export as Parquet to GCS Coldline/Archive storage ($0.004/GB/month). Create an external table if you need occasional access.
  3. Use table snapshots — take a snapshot before archiving, then delete original data. Snapshot is read-only and cheap.
  4. Partition expiration — auto-delete old partitions:
-- Auto-delete partitions older than 365 days
ALTER TABLE project.dataset.events
SET OPTIONS (partition_expiration_days = 365);

-- Export to GCS before deleting
EXPORT DATA OPTIONS (
  uri = 'gs://archive-bucket/events/year=2024/*.parquet',
  format = 'PARQUET', compression = 'SNAPPY'
) AS
SELECT * FROM events WHERE event_date < '2025-01-01';
7How do you handle late-arriving events?▼

Late-arriving events = events with an event timestamp earlier than the current processing window. Strategies:

  • Use column-based partitioning on event_time — data goes to the correct historical partition, not "today's" partition
  • Reprocess window — always reprocess last N days (e.g., 3 days) in your ETL to catch late arrivals
  • MERGE pattern — upsert late events into the curated table, updating existing records or inserting new ones
  • Two-layer architecture:
    • raw table — append-only, all events land here
    • curated table — rebuilt daily with dedup, covering a reprocessing window
  • Watermark — define a "late threshold" (e.g., 72 hours). Events beyond the watermark are dropped or routed to a dead-letter table.
8How would you migrate from Hive to BigQuery?▼

Migration approach:

  1. Assessment — inventory Hive tables, sizes, partition schemes, HQL scripts, dependencies
  2. Schema mapping:
    HiveBigQuery
    DatabaseDataset

    BucketClustering

  3. Data transfer:
    • Export Hive tables as Parquet/Avro to GCS (via distcp or Dataproc)

Must-Know BigQuery SQL Tricks

1Difference between WHERE and QUALIFY▼
WHEREQUALIFY
FiltersRow values (before aggregation/windowing)Window function results (after windowing)
Execution orderEarly — before GROUP BY and window functionsLate — after window functions are computed
Can useColumn values, expressions, subqueriesWindow functions: ROW_NUMBER, RANK, SUM OVER, etc.
-- WHERE: filter rows before windowing
SELECT * FROM events WHERE event_type = 'purchase';

-- QUALIFY: filter AFTER window function runs
SELECT * FROM events
QUALIFY ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) = 1;

-- Combined: WHERE filters first, then QUALIFY deduplicates
SELECT * FROM events
WHERE event_type = 'purchase'
QUALIFY ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) = 1;
2Use of TABLESAMPLE▼

TABLESAMPLE reads a random subset of data, reducing bytes scanned and cost. Great for exploratory queries on huge tables.

-- Sample ~10% of the table
SELECT * FROM `big_table` TABLESAMPLE SYSTEM (10 PERCENT);

-- Sample for quick profiling
SELECT
  COUNT(*) AS sample_rows,
  AVG(amount) AS avg_amount,
  MIN(event_date) AS min_date,
  MAX(event_date) AS max_date
FROM `project.dataset.events` TABLESAMPLE SYSTEM (1 PERCENT);

Important: TABLESAMPLE operates at the storage block level, not row level. The actual percentage may vary slightly. Not suitable for exact random samples — use ORDER BY RAND() LIMIT n for that (but it scans the full table).

3How to detect data skew▼

Data skew = uneven distribution of data across partitions or join keys, causing some workers to process disproportionately more data.

-- Detect skew on partition sizes
SELECT partition_id, total_rows, total_logical_bytes
FROM `project.dataset.INFORMATION_SCHEMA.PARTITIONS`
WHERE table_name = 'my_table'
ORDER BY total_logical_bytes DESC;

-- Detect skew on join key distribution
SELECT
  join_key,
  COUNT(*) AS row_count,
  COUNT(*) * 100.0 / SUM(COUNT(*)) OVER() AS pct
FROM my_table
GROUP BY join_key
ORDER BY row_count DESC
LIMIT 20;

-- Check for NULL key skew (NULLs all hash to same bucket)
SELECT COUNT(*) AS null_rows FROM my_table WHERE join_key IS NULL;

Signs in execution plan: One stage takes much longer than others. One worker processes 10x more data.

Fix: Filter out NULLs before JOINs, split hot keys into separate processing, or use APPROX functions.

4How to flatten nested fields▼
-- Access STRUCT fields directly with dot notation
SELECT
  order_id,
  customer.name AS customer_name,      -- STRUCT field
  customer.address.city AS city         -- nested STRUCT
FROM orders;

-- Flatten ARRAY of STRUCT with UNNEST
SELECT
  o.order_id,
  item.product_name,
  item.quantity,
  item.unit_price
FROM orders o,
UNNEST(o.line_items) AS item;           -- ARRAY<STRUCT>

-- Flatten ARRAY of primitives with OFFSET
SELECT
  id,
  tag,
  offset_idx
FROM my_table,
UNNEST(tags) AS tag WITH OFFSET AS offset_idx;

-- Re-aggregate after flattening
SELECT
  o.order_id,
  SUM(item.quantity * item.unit_price) AS total
FROM orders o, UNNEST(o.line_items) AS item
GROUP BY o.order_id;
5Use of WITH OFFSET▼

WITH OFFSET gives you the index position of each element when unnesting an array.

-- Get array elements with their index
SELECT
  id,
  element,
  idx
FROM my_table,
UNNEST(my_array) AS element WITH OFFSET AS idx
ORDER BY id, idx;

-- Example: first and last element of an array
SELECT
  id,
  arr[OFFSET(0)] AS first_element,              -- 0-based index
  arr[ORDINAL(ARRAY_LENGTH(arr))] AS last_element -- 1-based index
FROM my_table;

-- Filter by position: get only the 2nd element
SELECT id, element
FROM my_table, UNNEST(my_array) AS element WITH OFFSET AS idx
WHERE idx = 1;  -- 0-based → idx=1 is 2nd element
6How to generate sequences in BigQuery▼

BigQuery has no generate_series(), but you can use UNNEST + GENERATE_ARRAY:

-- Generate numbers 1 to 100
SELECT num FROM UNNEST(GENERATE_ARRAY(1, 100)) AS num;

-- Generate date sequence
SELECT date
FROM UNNEST(
  GENERATE_DATE_ARRAY('2026-01-01', '2026-12-31', INTERVAL 1 DAY)
) AS date;

-- Generate timestamp sequence (every hour for a day)
SELECT ts
FROM UNNEST(
  GENERATE_TIMESTAMP_ARRAY(
    '2026-04-22 00:00:00 UTC',
    '2026-04-22 23:00:00 UTC',
    INTERVAL 1 HOUR
  )
) AS ts;

-- Practical: fill date gaps in a report
WITH dates AS (
  SELECT date FROM UNNEST(GENERATE_DATE_ARRAY('2026-01-01', CURRENT_DATE())) AS date
)
SELECT d.date, COALESCE(SUM(o.amount), 0) AS daily_total
FROM dates d
LEFT JOIN orders o ON o.order_date = d.date
GROUP BY d.date ORDER BY d.date;

Top 10 BigQuery Interview Questions (High ROI)

If you're short on time, master these. Each links to the detailed answer above.

1Partitioning vs Clustering — Quick Summary▼

Partitioning = physically separates data (1 column, DATE/INT). Clustering = sorts data within partitions (up to 4 columns, any type). Use both together: partition by date, cluster by ID/category. Partition = exact pruning (visible in dry run). Clustering = block pruning (savings appear at runtime).

2Cost Optimization — Top 5 Levers▼
  1. Avoid SELECT * — name only needed columns
  2. Always filter on partition column
  3. Use clustering for frequently filtered columns
  4. Use materialized views for repeated aggregations
  5. Set maximum_bytes_billed as a safety net
3Window Functions — Key Patterns▼

ROW_NUMBER() for dedup, RANK()/DENSE_RANK() for rankings, LAG()/LEAD() for previous/next row, SUM() OVER() for running totals. Use QUALIFY to filter window results without subqueries.

4STRUCT / ARRAY — When and How▼

Use STRUCT for grouping fields (address, metadata). Use ARRAY for one-to-many (order items, tags). UNNEST() to flatten arrays. LEFT JOIN UNNEST to keep rows with empty arrays. ARRAY of STRUCT is the most common nested pattern in BigQuery.

5ANY_VALUE() — Avoid GROUP BY Bloat▼

Use when grouping by a key and a non-aggregated column is functionally dependent (same value per group). Avoids adding columns to GROUP BY which can increase shuffle cost. Example: SELECT user_id, ANY_VALUE(user_name), COUNT(*) FROM t GROUP BY user_id

6Streaming Buffer — Key Facts▼

Data queryable immediately after streaming. Buffer lasts ~90 minutes. DML can't modify buffer data. Buffer data excluded from time travel. Storage Write API (committed mode) is the modern replacement — exactly-once, no buffer limitations.

7CLONE vs COPY — Quick Comparison▼

CLONE: instant, zero-copy (metadata only), same region only, supports time-travel cloning. COPY: full data duplication, can cross regions, proportional to table size. Always prefer CLONE for same-region copies.

8Slot Model — How Compute Works▼

1 slot ≈ 1 vCPU + memory. On-demand: shared pool (~2,000 slots). Editions: baseline + autoscaling. Monitor via INFORMATION_SCHEMA.JOBS (total_slot_ms) and Cloud Monitoring. Slot contention = queries queuing/slowing. Fix: reserve dedicated slots or optimize queries.

9Denormalization Strategy▼

BigQuery favors wide, denormalized tables. JOINs are expensive (shuffle). Storage is cheap ($0.02/GB). Use nested fields (STRUCT + ARRAY) instead of normalized star schemas. Keep small, frequently-changing dimensions normalized. Pre-join everything else into fact tables.

10Query Execution Plan Analysis▼

Check in BigQuery UI after running. Key metrics: bytes scanned (cost), slot time vs wall time (parallelism), shuffle bytes (join/agg efficiency), spill to disk (memory pressure). Query INFORMATION_SCHEMA.JOBS for programmatic analysis. High shuffle + low output = good filter; high shuffle + high output = possible cross-join.

SQL Recipe: MERGE Operation

The MERGE statement combines INSERT, UPDATE, and DELETE in a single atomic operation. Use it for incremental loads and SCD processing.

BigQuery SQL — Merge 
MERGE target t
USING source s
ON t.window = s.window AND t.key = s.key

WHEN MATCHED THEN
  UPDATE SET total = s.total

WHEN NOT MATCHED THEN
  INSERT (window, key, total)
  VALUES (s.window, s.key, s.total);
➕

SQL Recipe: Insert Only Missing Records

Use NOT EXISTS to insert rows from source that don't already exist in the target. Avoids duplicates without a full MERGE.

BigQuery SQL — Insert Missing 
INSERT INTO `prj-d-prj.dw.target_table`
SELECT *
FROM `prj-d-prj.dw.source_table` s
WHERE NOT EXISTS (
    SELECT 1
    FROM `prj-d-prj.dw.target_table` t
    WHERE t.acct_no = s.acct_no
);
💡
Performance: For large tables, ensure the join key (acct_no) is part of the clustering columns to minimize bytes scanned.

SQL Recipe: Upsert (Update + Insert)

The standard upsert pattern in BigQuery uses MERGE to update existing records and insert new ones in a single DML statement.

BigQuery SQL — Upsert 
MERGE `project.dataset.target_table` T
USING `project.dataset.source_table` S
ON T.message_id = S.message_id

-- Update existing records
WHEN MATCHED THEN
  UPDATE SET
    T.last_updated_ts = CURRENT_TIMESTAMP(),
    T.col1 = S.col1,
    T.col2 = S.col2

-- Insert new records
WHEN NOT MATCHED THEN
  INSERT (message_id, col1, col2, last_updated_ts)
  VALUES (S.message_id, S.col1, S.col2, CURRENT_TIMESTAMP());
ℹ️
DML Quotas: BigQuery allows up to 20 DML statements per table per day (for free tier) and 1,500 for on-demand. Plan your MERGE frequency accordingly.

SQL Recipe: Delete Duplicates (Keep Oldest)

First detect duplicates, then delete all but the oldest record per key using ROW_NUMBER().

Step 1 — Detect Duplicates 
SELECT *
FROM `prj-d-panfinregstr.code.fin_linkhub_upstream_events`
QUALIFY COUNT(*) OVER (PARTITION BY message_id) > 1;
Step 2 — Delete Duplicates, Keep Oldest 
DELETE FROM `prj-d-panfinregstr.code.fin_linkhub_upstream_events`
WHERE message_id IN (
  SELECT message_id
  FROM (
    SELECT
      message_id,
      created_ts,
      ROW_NUMBER() OVER (
        PARTITION BY message_id
        ORDER BY created_ts ASC   -- oldest = 1
      ) AS rn
    FROM `prj-d-panfinregstr.code.fin_linkhub_upstream_events`
  )
  WHERE rn > 1   -- delete all except oldest
);
⚠️
Caution: This DELETE scans the table twice. For very large tables, consider creating a deduplicated copy with CREATE OR REPLACE TABLE ... AS SELECT ... QUALIFY ROW_NUMBER() ... = 1 instead.
