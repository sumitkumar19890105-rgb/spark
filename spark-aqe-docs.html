00 — CORE CONCEPT

How AQE Works

Traditional Spark plans queries entirely upfront based on estimated statistics. AQE breaks the query into stages separated by shuffles, and at each boundary it inspects actual runtime statistics before deciding the next move.

💡 Key Insight

AQE doesn't change how Spark runs — it changes what plan Spark runs, mid-execution, using real data instead of guesses.

Stage 1 Executes ↓ Map Output Statistics collected (actual partition sizes, row counts) ↓ AQE re-optimization rules fire: ├─ CoalesceShufflePartitions ├─ OptimizeSkewedJoin ├─ OptimizeLocalShuffleReader └─ DemoteBroadcastHashJoin ↓ Rewritten plan submitted for Stage 2 ↓ Stage 2 Executes (optimally) ✓
01 — FEATURE

Dynamic Partition Coalescing

The static spark.sql.shuffle.partitions setting creates a fixed number of output partitions. When data is uneven, you end up with either thousands of tiny tasks or a few massive ones.

⚠️ The Problem

Default 200 partitions for all queries — tiny queries get 200 nearly-empty tasks; large queries get 200 overloaded tasks. Either way, performance suffers.

Before & After

Before Coalescing (200 partitions, target = 128 MB): [2MB][3MB][1MB][150MB][2MB][4MB][148MB][1MB][3MB]... After AQE Coalescing: [ 6MB merged ][ 150MB ][ 154MB ]... (3 tasks, right-sized)

Configuration

python / pyspark 
# Enable partition coalescing
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")

# Target size per coalesced partition
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "128MB")

# Safety floor — never go below this count
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionNum", "1")

# Start high — AQE will reduce it dynamically
spark.conf.set("spark.sql.shuffle.partitions", "1000")

How the Merging Works

1
Stage completes

Spark collects map output sizes from all shuffle map tasks.

2
Statistics read

AQE inspects actual byte size of every shuffle partition.

3
Greedy bin-packing

Adjacent partitions merged left-to-right until advisoryPartitionSizeInBytes is reached.

4
Next stage launches

With far fewer, right-sized partitions. No re-shuffle needed.

✅ Tuning Tip

Set spark.sql.shuffle.partitions high (500–2000). AQE will coalesce down. Starting too low means AQE has no room to consolidate.

02 — FEATURE

Dynamic Join Strategy Switching

Spark picks a join strategy before execution based on estimated table sizes. If those estimates are wrong — stale stats, no ANALYZE TABLE run — the wrong strategy executes. AQE fixes this at runtime.

Join Strategy Comparison

StrategyWhen UsedShuffle?Memory
Broadcast Hash JoinOne side fits in executor memory❌ NoHigh (all executors)
Sort-Merge JoinBoth sides large✅ Both sidesModerate
Shuffle Hash JoinMedium tables✅ One sideModerate

Runtime Switching Flow

Pre-execution plan: SortMergeJoin (estimated left=500MB, right=800MB) ↓ Stage 1 (left side) materializes ↓ AQE checks actual size: 6 MB ← much smaller! ↓ DemoteBroadcastHashJoin rule fires ↓ Plan rewritten → BroadcastHashJoin ✓ (no sort, no merge, no re-shuffle)

Configuration

python / pyspark 
# AQE's runtime broadcast threshold
spark.conf.set("spark.sql.adaptive.autoBroadcastJoinThreshold", "30MB")

# Allow local shuffle read optimization
spark.conf.set("spark.sql.adaptive.localShuffleReader.enabled", "true")

# Static threshold (pre-execution — set conservatively)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10MB")
🔍 Verify in Spark UI

Look for *(AQE) prefix on plan nodes in the SQL tab. The physical plan will show BroadcastHashJoin even if the optimizer originally chose SortMergeJoin.

03 — FEATURE

Skew Join Optimization

Data skew is the #1 cause of slow Spark jobs. When one partition key dominates, one task processes gigabytes while others finish in seconds — the entire stage waits on that one straggler.

⚠️ Classic Symptom

Stage progress bar stuck at 199/200. One task has been running for 2 hours. All others finished in 3 minutes. This is skew.

Skew Detection Formula

skewThreshold = MAX( skewedPartitionFactor × median_partition_size, skewedPartitionThresholdInBytes ) Example: Median size = 64 MB skewedPartitionFactor = 5 skewedPartitionThreshold = 256 MB skewThreshold = MAX(5 × 64MB, 256MB) = MAX(320MB, 256MB) = 320 MB → Any partition > 320 MB is SKEWED

How AQE Fixes It

BEFORE (SortMergeJoin, skewed): Left: Right: [Part-0: 2 MB ] ──────────── [Part-0: 2 MB ] [Part-1: 3 MB ] ──────────── [Part-1: 3 MB ] [Part-2: 98 GB ] ──────────── [Part-2: 95 GB ] ← 1 straggler task! [Part-3: 1 MB ] ──────────── [Part-3: 1 MB ] AFTER (AQE splits + replicates): Left: Right: [Part-0: 2 MB ] ──────────── [Part-0: 2 MB ] [Part-1: 3 MB ] ──────────── [Part-1: 3 MB ] [Part-2a: 49 GB] ──────────┐ [Part-2: 95 GB ] (replicated copy 1) [Part-2b: 49 GB] ─────────┐└─ [Part-2: 95 GB ] (replicated copy 2) [Part-3: 1 MB ] ──────────── [Part-3: 1 MB ] → 2 parallel tasks instead of 1 monster task ✓

Configuration

python / pyspark 
# Enable skew join handling
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# How many × the median before a partition is "skewed"
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")

# Minimum absolute size to be considered skewed
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB")

# Target size per skew sub-partition (reuses advisory size)
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "64MB")

Algorithm Step-by-Step

1
Both shuffle stages complete

AQE has full partition size data for both sides of the join.

2
Median computed

Separately for each join side. Threshold calculated using the formula above.

3
Skewed partitions identified

Any partition on either side exceeding the threshold is flagged.

4
Split & replicate

Skewed partition split into N sub-partitions (size ÷ advisorySize). Matching partition on the other side replicated N times.

5
Join plan rewritten

N parallel tasks execute instead of 1 straggler. Stage completes in fraction of original time.

⚠️ Limitation

Skew optimization only applies to Sort-Merge Joins. It does not work on Broadcast Hash Joins (those have no shuffle to split).

04 — FEATURE

Dynamic Partition Pruning

DPP avoids scanning irrelevant partitions of a large fact table by pushing filter results from a dimension table at runtime — a critical optimization for star-schema / data warehouse queries.

Example Query

sql 
SELECT f.*, d.region
FROM   fact_sales f
JOIN   dim_date d ON f.date_id = d.date_id
WHERE  d.year = 2024

-- Without DPP: full scan of fact_sales (100s of partitions)
-- With DPP:    only 2024 partitions of fact_sales scanned ✓

How DPP Works

1. Filter dim_date WHERE year = 2024 → produces set of matching date_ids: {20240101, 20240102, ...} ↓ 2. Result injected as a runtime subquery filter into fact_sales scan ↓ 3. Only fact_sales partitions WHERE date_id IN (20240101, ...) read ↓ 4. I/O reduced from 100% → ~8% of the table ✓

Configuration

python / pyspark 
# Enable Dynamic Partition Pruning
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "true")

# Reuse broadcast result for pruning (avoids re-scanning dim table)
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.reuseBroadcastOnly", "true")

Requirements for DPP to Activate

🗂️

Fact table must be partitioned

DPP prunes physical partitions on disk. Unpartitioned tables see no benefit.

📡

Dimension side uses broadcast

DPP reuses the broadcast result to avoid re-scanning the dimension table.

🔑

Join key = partition key

The join column on the fact table must match the partition column.

🎯

Filter on dimension side

The WHERE clause must filter the dimension, not the fact table directly.

05 — INTERNALS

AQE Internals & Rule Engine

AQE wraps the physical plan in an AdaptiveSparkPlanExec node and creates QueryStageExec boundaries at every shuffle and broadcast exchange.

Plan Wrapping Flow

LogicalPlan ↓ (standard Catalyst optimization) SparkPlan ↓ InsertAdaptiveSparkPlan rule ↓ AdaptiveSparkPlanExec (root) ├── ShuffleQueryStageExec (boundary 1) ├── ShuffleQueryStageExec (boundary 2) └── BroadcastQueryStageExec (boundary 3) Each stage materializes independently. After each: re-optimization rules fire on the remainder.

Re-optimization Rules

RuleTriggers WhenAction
CoalesceShufflePartitionsAfter any shuffle stageMerges small adjacent partitions
OptimizeSkewedJoinBefore a SMJ with skewed inputSplits + replicates skewed partitions
OptimizeLocalShuffleReaderAfter broadcast decisionAvoids network shuffle via local read
DemoteBroadcastHashJoinActual size > thresholdReverts BHJ → SMJ to prevent OOM
🔬 Note

AQE only activates at shuffle/broadcast boundaries. Single-stage queries (no shuffles) receive no AQE benefit. Consider adding a repartition() to introduce a stage boundary if needed.

07 — REFERENCE

AQE vs. Manual Optimization

ScenarioWithout AQEWith AQE
Skewed join Manually salt keys, add SKEW HINT, rewrite ETL ✅ Auto-detected & split at runtime
Wrong partition count Tune shuffle.partitions per query per cluster size ✅ Auto-coalesced based on actual data
Bad join decision Force with BROADCAST hint, risk OOM ✅ Runtime switch if estimates were wrong
Partition pruning Must partition table AND write DPP-aware SQL ✅ Automatic with DPP flag + correct schema
Variable data sizes Re-tune configs every time data grows ✅ Self-adjusting every query
✅ Summary

AQE is Spark's self-healing optimizer. The less you know about your data distribution upfront, the more value AQE provides. Enable it, set a high shuffle partition count, and let runtime statistics do the work.
