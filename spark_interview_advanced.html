1. Spark Execution Model (Driver, Executor, DAG, Stages, Tasks)¶

Q: Explain the Spark execution model end-to-end.

Architecture Components:

  • Driver: The JVM process that runs main(). It creates the SparkContext, builds the DAG of transformations, splits it into stages, and schedules tasks on executors.
  • Executor: Worker JVM processes on cluster nodes. Each executor runs tasks in threads, stores cached data, and reports heartbeats to the driver.
  • DAG (Directed Acyclic Graph): When you call an action (count(), collect(), write()), Spark builds a DAG of all transformations leading to that action.
  • Stage: The DAG is split at shuffle boundaries (wide dependencies like groupBy, join, repartition). Each stage contains only narrow transformations (map, filter, union).
  • Task: Each stage is divided into tasks — one task per partition. Tasks are the smallest unit of work sent to executors.

Execution Flow:

User Code → SparkSession → Logical Plan → Catalyst Optimizer → Physical Plan
→ DAG Scheduler (splits into Stages) → Task Scheduler (creates Tasks per partition)
→ Tasks sent to Executors → Results returned to Driver

Key Insight: A job with 3 shuffles = 4 stages. Each stage runs in parallel across partitions.

2. Catalyst Optimizer Internals¶

Q: How does Catalyst Optimizer work internally?

Catalyst is Spark SQL's rule-based and cost-based query optimizer. It transforms your query through 4 phases:

Phase Description
1. Parsing SQL/DataFrame API → Unresolved Logical Plan (AST)
2. Analysis Resolves column names, table names, types using the Catalog → Resolved Logical Plan
3. Logical Optimization Applies rules: predicate pushdown, constant folding, column pruning, boolean simplification → Optimized Logical Plan
4. Physical Planning Generates one or more physical plans, picks the best using cost model → Selected Physical Plan

Key Optimization Rules:

  • Predicate Pushdown: push WHERE clauses closer to the data source
  • Column Pruning: only read required columns from Parquet/ORC
  • Constant Folding: evaluate 1 + 2 at compile time → 3
  • Boolean Simplification: x AND true → x
  • Join Reordering: smaller tables joined first (cost-based)

3. Tungsten Engine¶

Q: What is Tungsten and how does it improve performance?

Tungsten is Spark's execution engine that operates below the Catalyst optimizer. It focuses on CPU and memory efficiency.

Key Improvements:

Feature Description
Off-Heap Memory Manages memory directly (bypasses JVM GC) using sun.misc.Unsafe
Binary Data Format Stores data in compact binary format instead of Java objects (avoids serialization overhead)
Whole-Stage CodeGen Fuses multiple operators (filter, project, agg) into a single Java function — eliminates virtual function calls between operators
Cache-Aware Computation Designs data structures for CPU L1/L2 cache locality

Whole-Stage CodeGen is the most impactful feature:

  • Without it: each row passes through separate Filter.next() → Project.next() → Agg.next() (virtual calls)
  • With it: one tight loop processes filter+project+agg in a single generated method

Look for WholeStageCodegen and * markers in explain() output. The * prefix indicates operators fused into a single codegen stage.

5. How Spark Decides Number of Stages¶

Q: How does Spark decide the number of stages?

Spark splits the DAG into stages at shuffle boundaries (wide dependencies).

Narrow Transformations (no shuffle → same stage): map, filter, flatMap, mapPartitions, union, select, withColumn

Wide Transformations (cause shuffle → new stage boundary): groupBy, reduceByKey, join, repartition, distinct, sort, orderBy

Formula: Number of Stages = Number of Shuffles + 1

Example:

df.filter(...)           # Stage 0 (narrow)
  .groupBy(...).count()  # ← Shuffle → Stage 1
  .join(df2, ...)        # ← Shuffle → Stage 2  
  .orderBy(...)          # ← Shuffle → Stage 3

→ 4 stages (3 shuffles + 1)

# Stage boundaries demo — RDD API shows stages clearly with toDebugString()
rdd = sc.parallelize(range(100), 4)

# Stage 0: map (narrow)
# Shuffle boundary: reduceByKey
# Stage 1: reduceByKey output
result_rdd = rdd.map(lambda x: (x % 5, x)) \
                .reduceByKey(lambda a, b: a + b) \
                .sortByKey()

# toDebugString shows the stage breakdown and shuffle dependencies
print(result_rdd.toDebugString().decode("utf-8"))
print("\n→ Each indentation level = a stage boundary (shuffle)")

# DataFrame equivalent
df1 = spark.createDataFrame([(i, f"dept_{i%5}", i*10) for i in range(100)], ["id", "dept", "val"])
df2 = spark.createDataFrame([(f"dept_{i}", f"Dept Name {i}") for i in range(5)], ["dept", "dept_name"])

# This job has: scan → filter (Stage 0) → groupBy shuffle (Stage 1) → join shuffle (Stage 2) → sort shuffle (Stage 3)
multi_stage = df1.filter(F.col("val") > 200) \
                 .groupBy("dept").agg(F.sum("val").alias("total")) \
                 .join(df2, "dept") \
                 .orderBy("total")

multi_stage.explain()
print(f"\nCount Exchange nodes in the plan above = number of shuffles = stages - 1")

6. Optimizing a Slow Spark Job¶

Q: How do you optimize a slow Spark job?

Systematic Approach (the "SPARK" framework):

Step Action
S — Shuffle Reduce shuffles: use reduceByKey over groupByKey, avoid unnecessary repartition
P — Partitions Right-size spark.sql.shuffle.partitions (default 200 is too much for small data, too little for TB-scale)
A — Architecture Use broadcast joins for small tables, avoid collect() on large data
R — Read Enable predicate pushdown, column pruning, use columnar formats (Parquet/ORC)
K — Kache Cache/persist DataFrames used multiple times; unpersist when done

Common Antipatterns:

  1. collect() on large datasets → OOM on driver
  2. groupByKey() instead of reduceByKey() → moves all data before reducing
  3. Cartesian joins (cross joins) → N×M explosion
  4. Too many small files → excessive task overhead
  5. UDFs when built-in functions exist → loses Catalyst optimizations
  6. Not using AQE (Adaptive Query Execution) → misses runtime optimizations
In [ ]:
# BAD vs GOOD: Optimization examples

# ========================
# BAD: groupByKey (shuffles ALL data, then reduces)
# ========================
rdd_bad = sc.parallelize([(k, v) for k in range(10) for v in range(1000)])
# rdd_bad.groupByKey().mapValues(sum)  # BAD — sends all values across network

# GOOD: reduceByKey (reduces locally first, then shuffles reduced values)
rdd_good = rdd_bad.reduceByKey(lambda a, b: a + b)  # map-side combine!
print("reduceByKey result:", rdd_good.take(3))

# ========================
# BAD: Using Python UDF (black box to Catalyst)
# ========================

@udf(returnType=StringType())
def upper_udf(s):
    return s.upper() if s else None

# BAD: df.withColumn("name_upper", upper_udf("name"))

# GOOD: Use built-in function (Catalyst can optimize)
df_opt = df.withColumn("name_upper", F.upper(F.col("name")))
df_opt.explain()  # Notice: no PythonUDF in the plan

# ========================
# BAD: collect() then process in Python
# ========================
# bad_list = big_df.collect()  # Brings all data to driver!!!

# GOOD: Process in Spark
result_good = df.filter(F.col("salary") > 500).agg(F.sum("salary"))
result_good.show()

7. Handling Data Skew¶

Q: What is data skew and how do you handle it?

Data skew occurs when a few partitions have disproportionately more data than others. This causes:

  • A few tasks run much longer than the rest ("straggler tasks")
  • Other executors sit idle waiting
  • Potential OOM on the overloaded executor

Techniques to Handle Skew:

Technique How It Works
Salting Append random prefix to skewed keys, aggregate in 2 passes (salted → unsalted)
AQE Skew Join spark.sql.adaptive.skewJoin.enabled=true — Spark 3.x auto-splits skewed partitions at runtime
Broadcast Join If one side is small, broadcast it to avoid shuffle entirely
Isolate + Union Separate skewed keys, process with broadcast, union back with non-skewed result
Custom Partitioner Redistribute data to spread hot keys across partitions

# Simulate data skew and fix with salting
import random

# Create skewed data: key "hot_key" has 90% of the data
skewed_data = [(("hot_key", random.randint(1, 100))) for _ in range(9000)] + \
              [((f"key_{i}", random.randint(1, 100))) for i in range(1000)]
df_skewed = spark.createDataFrame(skewed_data, ["key", "value"])

# Show the skew
print("=== Data Distribution (SKEWED) ===")
df_skewed.groupBy("key").count().orderBy(F.desc("count")).show(5)

# ==========================================
# FIX: Salting technique (2-pass aggregation)
# ==========================================
SALT_BUCKETS = 10

# Pass 1: Add salt, do partial aggregation
df_salted = df_skewed.withColumn("salt", (F.rand() * SALT_BUCKETS).cast("int")) \
    .withColumn("salted_key", F.concat(F.col("key"), F.lit("_"), F.col("salt")))

df_partial = df_salted.groupBy("key", "salted_key") \
    .agg(F.sum("value").alias("partial_sum"), F.count("*").alias("partial_count"))

# Pass 2: Remove salt, do final aggregation
df_final = df_partial.groupBy("key") \
    .agg(F.sum("partial_sum").alias("total"), F.sum("partial_count").alias("cnt"))

print("\n=== After Salting Fix ===")
df_final.orderBy(F.desc("cnt")).show(5)

# ==========================================
# AQE Skew Join (Spark 3.x automatic fix)
# ==========================================
print("\nAQE Skew Join Config:")
print(f"  spark.sql.adaptive.enabled = {spark.conf.get('spark.sql.adaptive.enabled')}")
print(f"  spark.sql.adaptive.skewJoin.enabled = {spark.conf.get('spark.sql.adaptive.skewJoin.enabled', 'true')}")
print("  → AQE automatically detects and splits skewed partitions at runtime!")

8. Broadcast Join vs Shuffle Join¶

Q: When to use broadcast joins vs shuffle joins?

Aspect Broadcast Hash Join Sort-Merge Join (Shuffle)
When One side is small (< autoBroadcastJoinThreshold, default 10MB) Both sides are large
How Small table is broadcast to all executors, joined locally Both sides are shuffled by join key, then sorted and merged
Shuffle No shuffle on the large side Full shuffle on both sides
Cost Network cost = small_table × num_executors Network cost = both tables shuffled
Memory Small table must fit in executor memory Handles arbitrarily large data (spills to disk)
Best For Dimension table joins, lookup tables Fact-to-fact joins, large-to-large

Join Strategy Selection Order:

  1. Broadcast Hash Join (if one side < 10MB or hint used)
  2. Shuffle Hash Join (if one side is much smaller, fits in memory)
  3. Sort-Merge Join (default for large-large joins)
  4. Broadcast Nested Loop Join (for non-equi joins with small table)
  5. Cartesian Product (cross join — last resort)

9. Tuning: Shuffle Partitions, Memory & Cores¶

Q: How to tune spark.sql.shuffle.partitions, executor memory/cores?

Shuffle Partitions¶

spark.sql.shuffle.partitions = 200  (default)
  • Too few partitions → each partition too large → OOM, long GC pauses
  • Too many partitions → excessive task overhead, small files
  • Rule of thumb: target 128MB–256MB per partition after shuffle
  • With AQE: spark.sql.adaptive.coalescePartitions.enabled=true auto-adjusts post-shuffle partitions

Executor Memory¶

spark.executor.memory = 4g        # JVM heap for execution + storage
spark.executor.memoryOverhead = 1g # Off-heap (PySpark, overhead)
Total container = memory + memoryOverhead

Executor Cores¶

spark.executor.cores = 5   # threads per executor
  • 5 cores per executor is the sweet spot (avoids HDFS throughput bottleneck)
  • More cores → more memory contention, GC overhead
  • Fewer cores → underutilized memory

Sizing Formula (for YARN cluster)¶

Total cluster: 100 cores, 400 GB RAM, 20 nodes
→ 5 cores/executor → 20 executors (1 per node)
→ Leave 1 core for OS/YARN → 4 cores/executor → still 20 executors
→ Memory: (400GB/20 executors) - overhead = ~18GB/executor
→ spark.executor.memory=16g, memoryOverhead=2g

10. Caching vs Persistence¶

Q: What is the difference between cache() and persist()? When to use each?

Feature cache() persist(storageLevel)
Storage MEMORY_ONLY (default) Choose: MEMORY_ONLY, MEMORY_AND_DISK, DISK_ONLY, MEMORY_ONLY_SER, OFF_HEAP
Serialization Deserialized Java objects Optionally serialized (saves memory, costs CPU)
When use Quick default caching Fine-grained control over storage
Eviction LRU eviction if memory full → recomputation Depends on level (MEMORY_AND_DISK spills to disk)

When to Cache:

  • DataFrame is reused 2+ times in the DAG
  • Expensive computation (joins, aggregations) before branching
  • Iterative algorithms (ML training loops)

When NOT to Cache:

  • Used only once → waste of memory
  • Data is too large → eviction thrashing
  • Upstream data changes → stale cache

Important: Always unpersist() when done to free memory.

11. Shuffle Internals¶

Q: What happens during a shuffle?

A shuffle is the most expensive operation in Spark. It redistributes data across partitions.

Shuffle Write Phase (Map Side):

  1. Each task sorts/hashes its output by the target partition key
  2. Writes output to local disk as shuffle files (one file per reducer)
  3. Registers file locations with the Driver (MapOutputTracker)

Shuffle Read Phase (Reduce Side):

  1. Each reduce task fetches its data from ALL map tasks via network
  2. Data is pulled chunk-by-chunk (not all at once)
  3. Fetched data is merged/sorted before being processed

Shuffle Spill: When the in-memory sort buffer exceeds spark.shuffle.spill.initialMemoryThreshold, data spills to disk. Multiple spill files are later merged. This is a major performance killer.

How to Reduce Shuffle Cost:

  1. Use reduceByKey over groupByKey — map-side combine reduces data before shuffle
  2. Pre-partition data — if both DataFrames are partitioned by join key, no shuffle needed
  3. Broadcast small tables — eliminates shuffle for one side
  4. Use coalesce instead of repartition for reducing partitions (no shuffle)
  5. Filter early — less data to shuffle
  6. AQE — dynamically coalesces small shuffle partitions
In [ ]:
# groupByKey vs reduceByKey — shuffle cost comparison
rdd_pairs = sc.parallelize([(k % 50, 1) for k in range(100000)], 10)

# BAD: groupByKey shuffles ALL values, then aggregates
# Every (key, value) pair crosses the network
# grouped = rdd_pairs.groupByKey().mapValues(sum)

# GOOD: reduceByKey does map-side combine first (like a local pre-aggregation)
# Only partial sums cross the network
reduced = rdd_pairs.reduceByKey(lambda a, b: a + b)

print("=== reduceByKey vs groupByKey ===")
print(f"Input: {rdd_pairs.count()} records, {rdd_pairs.getNumPartitions()} partitions")
print(f"reduceByKey result: {reduced.count()} unique keys")
print(f"\ngroupByKey: shuffles {rdd_pairs.count()} records")
print(f"reduceByKey: shuffles ~{reduced.count()} records (pre-combined)")
print("→ reduceByKey reduces network I/O by orders of magnitude!")

# Show shuffle in DataFrame plan

12. Repartition vs Coalesce¶

Q: What is the difference between repartition() and coalesce()?

Aspect repartition(n) coalesce(n)
Shuffle Full shuffle — data redistributed No shuffle — partitions merged locally
Direction Increase or decrease partitions Only decrease partitions
Distribution Evenly distributed May produce uneven partitions

13. Partitioning Strategy for Large Datasets¶

Q: How do you design a partitioning strategy for large datasets?

Write-Time Partitioning (partitionBy)¶

df.write.partitionBy("year", "month").parquet("/data/events")
# Creates: /data/events/year=2025/month=01/part-00000.parquet
  • Enables partition pruning — only reads relevant directories
  • Choose columns with low cardinality (date, region, status — NOT user_id)

df.write.bucketBy(100, "user_id").sortBy("user_id").saveAsTable("bucketed_table")

  • Only works with Hive metastore tables
  • Best for columns used repeatedly in joins

# Skewed left table: "hot_key" appears 90% of the time

            [(f"key_{i}", i) for i in range(1000)]

# Right table (dimension — no skew)

             [(f"key_{i}", f"Category_{i}") for i in range(1000)]

# Step 1: Salt the left (skewed) table

    "salt", (F.rand() * SALT_BUCKETS).cast("int")
).withColumn(
    "salted_key", F.concat(F.col("key"), F.lit("_"), F.col("salt"))
)

    spark.range(SALT_BUCKETS).toDF("salt")
).withColumn(
    "salted_key", F.concat(F.col("key"), F.lit("_"), F.col("salt"))
)

# Step 3: Join on salted_key (skew is now spread across SALT_BUCKETS partitions)
result_salted = df_left_salted.join(
    df_right_exploded.select("salted_key", "category"),
    "salted_key"
).drop("salted_key", "salt")

print(f"Salted join result count: {result_salted.count()}")
result_salted.groupBy("key", "category").count().orderBy(F.desc("count")).show(5)

16. Spark Memory Model & OOM Debugging¶

Q: Explain the Spark memory model. What causes OOM and how to debug it?

Unified Memory Manager (Spark 1.6+)¶

Executor JVM Heap (spark.executor.memory)
├── Reserved Memory (300MB) — Spark internal
├── User Memory (40%) — user data structures, UDF variables, RDD metadata
└── Unified Memory (60%) — shared between:
    ├── Execution Memory — shuffles, joins, sorts, aggregations
    └── Storage Memory — cached DataFrames/RDDs
    (boundary is soft — they can borrow from each other)

Common OOM Causes¶

Cause Symptom Fix
collect() on large data Driver OOM Use .show(), .take(), or write to storage
Skewed data in groupBy/join Executor OOM on one task Salting, AQE skew join, broadcast
Too many partitions cached Storage memory full Unpersist, use MEMORY_AND_DISK
Large broadcast variable Driver OOM (broadcast built on driver) Reduce table size, increase spark.driver.maxResultSize
Exploding joins (many-to-many) Executor OOM Check join keys for duplicates, add filters
PySpark UDF holding large objects Off-heap (memoryOverhead) OOM Increase spark.executor.memoryOverhead

Spill to Disk¶

When execution memory is full during shuffle/sort/join, Spark spills intermediate data to disk. Visible in Spark UI → Stage Details → "Shuffle Spill (Disk)" and "Shuffle Spill (Memory)".

  3. Check: Shuffle spill metrics, task duration skew
  4. Increase: spark.executor.memory, spark.executor.memoryOverhead

18. Predicate Pushdown & Column Pruning¶

Q: What are predicate pushdown and column pruning? How does Spark optimize reads?

Predicate Pushdown¶

Spark pushes WHERE clause filters down to the data source so less data is read from disk/storage.

  • For Parquet/ORC: filters applied at file level using min/max statistics in file footer → entire row groups skipped
  • For JDBC: WHERE clause sent to the database → less data transferred
  • For Hive: filter pushed to Hive metastore

Column Pruning¶

Spark only reads the columns you actually use in your query.

  • For Parquet (columnar): only requested columns' data blocks are read from disk
  • For CSV/JSON (row-based): all columns read, then filtered in memory

How to Verify¶

In explain() output, look for:

  • PushedFilters: [IsNotNull(col), GreaterThan(col, 100)] — predicate pushdown
  • ReadSchema: struct<col1:int, col2:string> — column pruning (only needed columns listed)

19. Structured Streaming: Micro-batch, Watermark, Checkpoint¶

Q: Explain micro-batch vs continuous processing. How does exactly-once work?

Micro-Batch Processing (Default)¶

  • Spark processes data in small batches at regular intervals (trigger interval)
  • Each batch is a small Spark job with full DataFrame optimizations
  • Latency: 100ms–seconds

Continuous Processing (Experimental, Spark 2.3+)¶

  • Processes each record as it arrives — lower latency (~1ms)
  • Limited operator support (no aggregations)
  • Uses epoch-based checkpointing

Exactly-Once Semantics¶

Achieved through:

  1. Replayable sources: Kafka offsets, file-based sources can be replayed from a checkpoint
  2. Checkpointing: Stores offsets, state, and metadata for recovery
  3. Idempotent sinks: Writes must be idempotent (e.g., Parquet files with unique naming, Delta Lake transactions)
  4. Write-Ahead Log (WAL): For non-idempotent sinks

Watermarking (Late Data Handling)¶

df.withWatermark("event_time", "10 minutes")
  .groupBy(window("event_time", "5 minutes"))
  .count()
  • Watermark = current max event time - threshold
  • Data arriving before the watermark is included
  • Data arriving after the watermark is dropped
  • Allows Spark to clean up old state (bounded state for aggregations)

Checkpointing¶

query = df.writeStream \
    .outputMode("append") \
    .option("checkpointLocation", "/path/to/checkpoint") \
    .start()

Stores: source offsets, sink state, aggregation state → enables exactly-once recovery after failure.

21. Adaptive Query Execution (AQE)¶

Q: What is AQE and why is it a game-changer in Spark 3.x?

Adaptive Query Execution (AQE) is a Spark optimization framework that dynamically modifies the execution plan at runtime based on actual shuffle statistics. It improves performance by handling skewed joins, optimizing partition sizes, and dynamically changing join strategies.

Key Features:¶

Feature What It Does Config
Dynamic Coalescing Merges small post-shuffle partitions into larger ones spark.sql.adaptive.coalescePartitions.enabled
Dynamic Join Selection Switches from Sort-Merge Join to Broadcast Join at runtime if one side is small spark.sql.adaptive.autoBroadcastJoinThreshold
Skew Join Handling Detects and splits skewed partitions automatically spark.sql.adaptive.skewJoin.enabled

How It Works:¶

Compile Time Plan (Static)
        |
        v
    Stage 1 executes
        |
        v
    Shuffle statistics collected (partition sizes)
        |
        v
    AQE RE-OPTIMIZES remaining plan
        |
        v
    Stage 2 executes with BETTER plan

Before vs After AQE:¶

Problem Without AQE With AQE
200 shuffle partitions, only 10 have data 190 empty tasks wasted Auto-coalesced to ~10 partitions
Small table grows slightly > 10MB Forced Sort-Merge Join Detected at runtime → Broadcast
One key has 90% of data One task runs forever Split into multiple sub-tasks

Configs:¶

22. Bucketing¶

Q: What is bucketing and how does it eliminate shuffles?

Bucketing is a write-time optimization that pre-shuffles and pre-sorts data into a fixed number of buckets (files) based on a column's hash value. When you later join or aggregate on that column, Spark skips the shuffle entirely because data is already co-located.

Bucketing vs Partitioning:¶

Aspect Partitioning Bucketing
Splits by Distinct values (directories) Hash of column (fixed # files)
Best for Low-cardinality columns (date, region) High-cardinality columns (user_id, order_id)
Eliminates Full table scans (partition pruning) Shuffles on joins/aggregations
File structure /col=value/ directories Fixed number of files per partition
Number of outputs Varies (one dir per distinct value) Fixed (you specify bucket count)

How It Works:¶

WRITE TIME:
  Table A (bucketed by user_id, 100 buckets)
  +-- bucket-00000  ->  hash(user_id) % 100 == 0
  +-- bucket-00001  ->  hash(user_id) % 100 == 1
  +-- ...
  +-- bucket-00099  ->  hash(user_id) % 100 == 99

  Table B (bucketed by user_id, 100 buckets)
  +-- bucket-00000  ->  same hash!
  +-- ...
  +-- bucket-00099

JOIN TIME:
  bucket-00000(A) joins ONLY with bucket-00000(B)  <- NO SHUFFLE!
  bucket-00001(A) joins ONLY with bucket-00001(B)
  ...

Rules for Shuffle Elimination:¶

  1. Both tables must be bucketed on the same column
  2. Both must have the same number of buckets (or one is a multiple of the other)
  3. Join/group column must match the bucket column
  4. Data must be read from the bucketed table (not a temp view)

23. Serialization: Kryo vs Java¶

Q: What is serialization in Spark? Why use Kryo over Java serializer?

Serialization is the process of converting objects into byte streams for:

  • Shuffling data between executors (over network)
  • Caching data in serialized form (MEMORY_ONLY_SER)
  • Broadcasting variables to all executors
  • Closure serialization (sending functions to executors)

Java Serializer vs Kryo:¶

Aspect Java Serializer (Default) Kryo Serializer
Speed Slow ~10x faster
Size Large (verbose metadata) ~2-5x smaller
Compatibility Any class implementing Serializable Requires class registration for best perf
Ease Zero config Needs registration
Config Default spark.serializer=org.apache.spark.serializer.KryoSerializer

When Serialization Matters:¶

+-----------+     SERIALIZE      +-----------+     DESERIALIZE     +-----------+
|Executor 1 |  --- objects --->  | Network   |  --- bytes ------>  |Executor 2 |
|(map side) |     to bytes       | Transfer  |     to objects      |(reduce)   |
+-----------+                    +-----------+                     +-----------+

Smaller serialized size = Less network I/O = Less disk I/O (spills) = FASTER

Impact on Performance:¶

  • Shuffle: Every record crossing a partition boundary is serialized -> transferred -> deserialized
  • Cache (SER): MEMORY_ONLY_SER stores serialized bytes (saves memory, costs CPU)
  • Broadcast: Large lookup tables serialized once, sent to all executors
  • Task closure: Your lambda/function is serialized and sent to each executor

Common Serialization Errors:¶

org.apache.spark.SparkException: Task not serializable

Cause: Your transformation closure captures a non-serializable object (DB connection, SparkSession, etc.)

Fix: Move the object creation inside the transformation or use transient.

24. Delta Lake & Apache Iceberg¶

Q: What are Delta Lake and Iceberg? Why are they needed over plain Parquet?

Plain Parquet/ORC on a data lake has critical limitations: no ACID transactions, no schema enforcement, no time travel, and the small-file problem. Table formats like Delta Lake and Iceberg solve these.

Problems with Plain Parquet:¶

Problem Impact
No ACID transactions Concurrent writes corrupt data
No schema enforcement Bad data silently enters
No time travel Can't rollback mistakes
Small files accumulate Slow reads, high metadata overhead
No row-level updates/deletes Must rewrite entire partitions

Delta Lake vs Iceberg:¶

Feature Delta Lake Apache Iceberg
Developed by Databricks Netflix (now Apache)
Transaction Log _delta_log/ (JSON + Parquet checkpoints) metadata/ (Avro manifest files)
ACID Yes Yes
Time Travel VERSION AS OF, TIMESTAMP AS OF snapshot-id, as-of-timestamp
Schema Evolution Add, rename, reorder Full evolution including type widening
Partition Evolution Requires rewrite Hidden partitioning (change without rewrite)
Engine Support Spark, Databricks, Flink Spark, Flink, Trino, Hive, Dremio
Z-Ordering OPTIMIZE ZORDER BY Sort-order in metadata
Compaction OPTIMIZE command Rewrite manifests
Merge (Upsert) MERGE INTO MERGE INTO

Delta Lake Architecture:¶

Storage Layer (S3/GCS/ADLS):
+-- _delta_log/
|   +-- 00000000000000000000.json    <- Transaction 1 (add files)
|   +-- 00000000000000000001.json    <- Transaction 2 (delete + add)
|   +-- 00000000000000000010.checkpoint.parquet  <- Checkpoint every 10 txns
|   +-- ...
+-- part-00000-xxx.parquet           <- Actual data files
+-- part-00001-xxx.parquet
+-- ...

Each JSON log entry records:
  - Files added
  - Files removed
  - Schema changes
  - Commit metadata

Key Operations:¶

  • Time Travel: Query data as it existed at a past version/timestamp
  • MERGE (Upsert): INSERT + UPDATE + DELETE in one atomic operation
  • OPTIMIZE: Compact small files into larger ones
  • Z-ORDER: Co-locate related data for faster reads on multiple columns
  • VACUUM: Delete old files no longer referenced by any version

Output rows One row per group Same number of rows as input

26. Accumulator & Broadcast Variables¶

Q: What are shared variables in Spark? Explain Accumulators and Broadcast variables with pitfalls.

In distributed computing, Spark ships a copy of each variable to every task. For efficiency, Spark provides two types of shared variables:

Broadcast Variables:¶

Purpose: Efficiently distribute a large read-only lookup table to all executors once, rather than shipping it with every task.

WITHOUT Broadcast:                    WITH Broadcast:
Task 1 gets copy of lookup (100MB)    Driver sends lookup ONCE to each executor
Task 2 gets copy of lookup (100MB)    All tasks on that executor share it
Task 3 gets copy of lookup (100MB)
...                                   Memory: 100MB per EXECUTOR (not per task)
Memory: 100MB per TASK!

Use Cases:

  • Dimension/lookup tables for map-side joins
  • Configuration maps
  • ML model weights
Aspect Detail
Immutable Cannot be modified after broadcast
Serialized once Sent via BitTorrent-like protocol to executors
Lazy Only sent when first accessed on executor
Unpersist Call .unpersist() to free memory
Size limit Must fit in executor memory; typically < 1-2 GB

Accumulators:¶

Purpose: Aggregate values from executors back to the driver (counters, sums). Write-only from executor perspective; read-only on driver.

Critical Pitfall — Double Counting:

Task fails -> Spark retries it -> Accumulator incremented AGAIN!

Also: If a transformation with accumulator is recomputed (due to cache eviction
or lineage replay), the accumulator counts AGAIN.

Rules:

  1. Only use accumulators inside actions (not transformations) for guaranteed accuracy
  2. In transformations, accumulators are "best effort" — may double-count on retries
  3. Use AccumulatorV2 for custom types (Spark 2.0+)

27. Fault Tolerance & Checkpointing¶

Q: How does Spark handle fault tolerance? What is checkpointing and when to use it?

Spark's Fault Tolerance Mechanism:¶

Spark achieves fault tolerance through RDD Lineage — a record of all transformations used to build a dataset.

Lineage Graph:
  textFile("data.txt")
      |
  .filter(line => line.contains("error"))
      |
  .map(line => (line.split(" ")(0), 1))
      |
  .reduceByKey(_ + _)

If a partition is LOST (executor crash):
  Spark replays the lineage from the last reliable source
  Only the LOST partition is recomputed, not the entire dataset

Caching vs Checkpointing:¶

Aspect cache() / persist() checkpoint()
Storage Memory/disk on executors HDFS/GCS/S3 (reliable storage)
Lineage Preserved (can recompute if lost) Truncated (lineage cut)
Survives Only within the application Across application restarts
Speed Fast (executor-local) Slower (writes to distributed FS)
Use case Reuse within a job Long lineage chains, iterative algorithms
Fault tolerance Recompute from lineage if lost No recomputation needed

When to Checkpoint:¶

  1. Long lineage chains (iterative ML algorithms — 100s of iterations)
  2. Streaming (required for stateful operations, exactly-once)
  3. Wide dependencies (after expensive shuffles to avoid recomputation)
  4. Graph algorithms (PageRank, connected components)

Streaming Checkpointing:¶

In Structured Streaming, checkpointing stores:

  • Offset tracking (which messages have been consumed)
  • State store (aggregation state for stateful operations)
  • Metadata (query configuration)

This enables exactly-once semantics:

Source (Kafka) ---> Spark Processing ---> Sink (Delta/DB)
                         |
                    Checkpoint Dir
                    (offsets + state)

If failure occurs:
  1. Restart reads checkpoint to find last committed offset
  2. Reprocesses from that offset
  3. Idempotent sink ensures no duplicates

28. Small File Problem¶

Q: What is the small file problem in Spark? How do you fix it?

The Problem:¶

When Spark writes data, each task creates one output file. With high parallelism (e.g., 200 shuffle partitions) or frequent writes (streaming), you end up with thousands of tiny files.

BAD: 200 tasks writing 1KB each = 200 files of 1KB
     - HDFS NameNode overhead (150 bytes metadata per file)
     - Slow reads (high seek time, many file opens)
     - Poor compression ratio
     - Cloud storage list operations become expensive

GOOD: 4 files of ~50KB each
     - Efficient sequential reads
     - Better compression
     - Less metadata overhead

Impact:¶

Area Impact of Small Files
NameNode / Metadata Each file = ~150 bytes in NameNode memory; millions of files = OOM
Read Performance Each file requires a separate open/close; scheduling overhead per file
Cloud Storage (S3/GCS) LIST operations are expensive; throttling on many small reads
Downstream Jobs More tasks than needed; driver bottleneck collecting file list

Solutions:¶

Solution When to Use How
coalesce() before write Batch jobs with known output size df.coalesce(10).write.parquet(...)
repartition() before write Need even distribution df.repartition(10).write.parquet(...)
maxRecordsPerFile Control file size .option("maxRecordsPerFile", 1000000)
OPTIMIZE (Delta) Post-write compaction OPTIMIZE delta.'/path/' ZORDER BY (col)
AQE coalescing Automatic during shuffle spark.sql.adaptive.coalescePartitions.enabled=true
Streaming: trigger Control batch frequency trigger(processingTime="5 minutes")
Hive: concatenate Merge small files in Hive tables ALTER TABLE t CONCATENATE

29. Dynamic Resource Allocation¶

Q: What is Dynamic Resource Allocation? When and why do you use it?

Dynamic Resource Allocation (DRA) allows Spark to automatically scale the number of executors up or down based on workload demand. Instead of requesting a fixed number of executors for the entire job, Spark requests more when busy and releases idle ones back to the cluster.

Without vs With DRA:¶

WITHOUT DRA (Fixed):
  Job requests 100 executors at start
  Stage 1: Uses all 100 (heavy shuffle)
  Stage 2: Only uses 10 (light filter)     <- 90 executors IDLE but held!
  Stage 3: Needs 150                       <- Can't scale up!

WITH DRA (Dynamic):
  Stage 1: Scales UP to 100 executors
  Stage 2: Scales DOWN to 10 (releases 90 back to cluster)
  Stage 3: Scales UP to 150 (requests more from cluster)

How It Works:¶

        Executor Idle for > idle timeout
                    |
                    v
         Spark releases executor
                    |
                    v
    Resources returned to Cluster Manager (YARN/K8s)


        Pending tasks in scheduler queue
                    |
                    v
         Spark requests new executor
                    |
                    v
    Cluster Manager allocates new container

Key Configurations:¶

Config Default Description
spark.dynamicAllocation.enabled false Master switch
spark.dynamicAllocation.minExecutors 0 Minimum executors to keep
spark.dynamicAllocation.maxExecutors infinity Maximum executors allowed
spark.dynamicAllocation.initialExecutors minExecutors Starting count
spark.dynamicAllocation.executorIdleTimeout 60s Release idle executors after this
spark.dynamicAllocation.schedulerBacklogTimeout 1s Request new executors after tasks pending this long
spark.dynamicAllocation.cachedExecutorIdleTimeout infinity Timeout for executors with cached data

Requirements:¶

  1. External Shuffle Service must be enabled (spark.shuffle.service.enabled=true) — because executors holding shuffle data can't be killed
  2. On Kubernetes: Use shuffle tracking instead (spark.dynamicAllocation.shuffleTracking.enabled=true)

When to Use:¶

Scenario DRA? Why
Shared cluster (YARN/K8s) Yes Don't hog resources when idle
Variable workload stages Yes Scale up for shuffles, down for filters
Streaming (micro-batch) Careful Can cause latency spikes from scaling
Fixed-size dedicated cluster No No benefit, adds overhead
Short-lived jobs (< 2 min) No Scaling overhead > benefit

30. Speculative Execution¶

Q: What is speculative execution? How does it handle straggler tasks?

Speculative execution (also called "backup tasks") is Spark's mechanism to handle straggler tasks — tasks that run significantly slower than others in the same stage, usually due to:

  • Hardware issues (slow disk, degraded node)
  • Data skew within a partition
  • GC pauses
  • Network congestion

How It Works:¶

Stage with 100 tasks:

Task 1:  ████████████ DONE (30s)
Task 2:  ████████████ DONE (28s)
Task 3:  ████████████ DONE (32s)
...
Task 99: ████████████ DONE (31s)
Task 50: ████████████████████████████████░░░ STILL RUNNING (90s) <- STRAGGLER!

Spark detects: Task 50 is 1.5x slower than median (median = 30s)

Action: Launch COPY of Task 50 on a different executor
        First one to finish wins; other is killed.

Task 50 (original):  ████████████████████████████████████████ (120s)
Task 50 (speculative): ████████████████████ DONE! (35s)  <- WINS!

Key Configurations:¶

Config Default Description
spark.speculation false Enable speculative execution
spark.speculation.interval 100ms How often to check for stragglers
spark.speculation.multiplier 1.5 Task is straggler if runtime > multiplier x median
spark.speculation.quantile 0.75 Start speculation after 75% of tasks complete
spark.speculation.minTaskRuntime 100ms Minimum runtime before a task can be speculated

When to Enable:¶

Scenario Enable? Reason
Large clusters with heterogeneous hardware Yes Slow nodes are common
Cloud environments (shared tenants) Yes Noisy neighbors cause variability
Production batch jobs Yes Reduces tail latency
Streaming jobs Careful Can increase resource usage
Jobs with side effects (write to DB) NO Duplicate writes!
Skewed data (known) No Fix skew instead (salting/AQE)

Pitfalls:¶

  1. Non-idempotent operations: If your task writes to an external DB, speculative copies can cause duplicates
  2. Resource waste: Speculative tasks consume extra CPU/memory
  3. False positives: A task may be slow due to data skew, not hardware — speculation won't help
  4. Accumulator double-counting: Speculative task's accumulator updates may be applied twice
